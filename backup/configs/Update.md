
## 1. ОБЩАЯ АРХИТЕКТУРА СИСТЕМЫ

Система представляет собой конвейер последовательной обработки запросов с использованием двух экземпляров одной и той же модели формата GGUF, распределённых по ролям. Обработка осуществляется в синхронном режиме без параллельного выполнения инференса для обеспечения максимальной вычислительной мощности каждого этапа. Ввод пользователя проходит через цепочку модулей аналитики, после чего формируется итоговый ответ с обязательной проверкой релевантности.

Архитектура разделена на три логических слоя:

### Слой А — Фактологическое ядро
Модуль генерирует краткий сухой ответ на основе внутренних знаний модели без расширения контекста и дополнительных рассуждений.

### Слой Б — Аналитический конвейер
Группа модулей выполняет поиск во внешних источниках, проверку на противоречия и этическую корректность перед формированием расширенного ответа.

### Слой В — Контроль качества
Система оценки метрик определяет соответствие итогового результата требованиям и запускает рекурсивное уточнение при необходимости.

## 2. НАСТРОЙКИ ИНФЕРЕНСА МОДЕЛЕЙ

Для обеих моделей используется формат чат-темплета ChatML с обязательными маркерами `<|system|>`, `<|user|>`, `<|assistant|>` согласно спецификации Qwen 2.5. Модель должна быть загружена в квантовании **Q6_K** или **Q8_0** для минимизации потери качества при малом размере весов. Оба экземпляра инициализируются с количеством потоков, равным общему количеству физических ядер процессора системы. Это обеспечивается за счёт последовательности выполнения задач — когда работает одна модель, вторая находится в состоянии ожидания, что исключает конкуренцию за ресурсы CPU.

### 2.1. GGUF основная (Фактологическое ядро)

| Параметр | Значение | Назначение |
|----------|----------|------------|
| Размер контекста | 4096 токенов | Хватает для запроса и краткого ответа |
| Температура генерации | 0.15 | Минимальная для точных фактов |
| Топ-п выборка | 0.90 | Узкая выборка вариантов |
| Максимальная длина | 256 токенов | Около 3 предложений |
| Штраф повторов | 1.1 | Предотвращение зацикливания |

### 2.2. GGUF уточнения (Аналитический модуль)

| Параметр | Значение | Назначение |
|----------|----------|------------|
| Размер контекста | 8192 токенов | Запрос, факты и сниппеты поиска |
| Температура генерации | 0.55 | Баланс креативности и точности |
| Топ-п выборка | 0.95 | Расширенная выборка для рассуждений |
| Максимальная длина | 512 токенов | Около 7 предложений |
| Штраф повторов | 1.15 | Строже контроль повторов |

Обе модели используют один и тот же файл весов `.gguf`, различаясь только системными инструкциями внутри промта.

## 3. СЕКВЕНЦИЯ ВЫПОЛНЕНИЯ МОДУЛЕЙ

Обработка запроса происходит в строго определённом порядке для гарантированного формирования корректного рекурсивного ответа. Порядок активации критичен для работы системы:

### Этап 1. Модуль извлечения фактов GGUF основная
Пользовательский запрос передаётся первому экземпляру модели для получения базовой фактологической информации. Этот этап ограничен по длине для предотвращения лишних рассуждений. Результат сохраняется как эталон внутренней информации модели.

### Этап 2. Модуль веб-поиска
На этом этапе система обращается к внешнему поисковому API. Рекомендуемые поставщики: **Yandex Search API** для максимального покрытия русскоязычного контента либо **Tavily API** на этапе разработки. Количество результатов ограничено 5 сниппетами, так как большее количество перегрузит контекст модели 0.5B. Сниппеты проходят предварительную фильтрацию — удаляются коммерческие предложения, реклама и дубликаты. Каждому результату присваивается порядковый номер для последующих ссылок.

### Этап 3. Модуль проверки противоречий
Полученные факты сравниваются с данными поиска. Используются семантические методы сравнения для выявления расхождений в ключевых утверждениях. При обнаружении противоречий система фиксирует их наличие и формирует метку о приоритете данных поиска при формулировании финального ответа. Эта проверка выполняется до передачи данных в Модель Б.

### Этап 4. Модуль этической оценки
Тексты, включая результаты поиска, проверяются на соответствие этическим нормам. Для малой модели рекомендуется использовать внешний список правил на Python (ключевые слова по категориям вреда), а не полагаться на встроенную оценку самой модели, которая может давать галлюцинации. Результаты сохраняются как флаг наличия нарушений.

### Этап 5. Модуль расширения контекста GGUF уточнения
GGUF уточнения получает полный пакет данных: исходный запрос, факты от GGUF основной, результаты поиска, метки о противоречиях и этических нарушениях. На основе этого контекста формируется развёрнутый ответ с концептуальным расширением темы.

### Этап 6. Модуль оценки релевантности
После генерации проводится вычисление косинусного сходства между вектором пользовательского запроса и вектором совокупного текста (факты + итоговый ответ). Используется эмбеддинг-модель, оптимизированная для русского языка, например `cointegrated/rubert-tiny2`. Результатом является числовой коэффициент от 0 до 1.

## 4. МЕТОДЫ ОЦЕНКИ КАЧЕСТВА И АНАЛИТИКА ВЫВОДА

Реализация аналитики вывода обязательна для контроля качества работы системы. Без метрик невозможно определить эффективность работы каждой ступени конвейера.

### 4.1. Метрики для отслеживания

| Метрика | Описание | Нормальное значение |
|---------|----------|---------------------|
| Семантический вес | Косинусное сходство запроса к ответу | ≥0.90 |
| Время генерации | Миллисекунды для каждого модуля | Зависит от CPU |
| Полезность поиска | Доля релевантных сниппетов | ≥0.70 |
| Противоречия | Количество выявленных конфликтов | 0–1 |
| Этика | Флаг наличия нарушений | Отсутствуют |
| Длина текста | Токены промежуточного и итогового текста | 256–512 |

### 4.2. Расчёт общей уверенности
Система рассчитывает интегральный показатель уверенности, который учитывает базовую релевантность, штрафы за противоречия и бонусы за отсутствие этических проблем. Минимально допустимый порог составляет 0.75, ниже которого выдаётся предупреждение пользователю.

### 4.3. Запись логов
Каждый этап обработки записывается в центральный лог-файл с метками времени, использованными параметрами и полученными метриками. Это позволяет анализировать производительность системы и оперативно выявлять деградацию качества.

## 5. ЛОГИКА РЕКУРСИВНОГО УТОЧНЕНИЯ

При низком показателе релевантности система запускает рекурсивный цикл повторной генерации с модификацией входных параметров. Максимальное количество попыток ограничено тремя для предотвращения бесконечных циклов.

### 5.1. Попытка 1: Стандартный режим
Выполняется полная обработка запроса по стандартному алгоритму. Если релевантность достигнута (≥0.90), ответ выдается без дальнейшего уточнения.

### 5.2. Попытка 2: Уточнённый промт
При значении от 0.75 до 0.90 система генерирует более строгий промт для Модели Б с указанием на необходимость сосредоточения на теме запроса и сокращения отступлений.

### 5.3. Попытка 3: Расширенные источники
При сохранении низкого результата увеличивается количество источников поиска до 7–10, добавляется требование использовать минимум три источника в ответе.

### 5.4. Финализация
Если после трёх попыток релевантность остаётся ниже 0.75, система выдает ответ со статусом «завершено с предупреждением». Пользователю сообщается о потенциальной неточности результата.

## 6. ПАТТЕРНЫ ФИЛЬТРАЦИИ И ПРЕДОБРАБОТКИ ДАННЫХ

### 6.1. Фильтрация результатов поиска
Перед передачей в модель все сниппеты подвергаются фильтрации по следующим критериям:

*   Удаляются ссылки, содержащие слова «купить», «реклама», «акция»
*   Оставляются только информационные и новостные домены
*   Длинные тексты обрезаются до 300 символов
*   Повторяющиеся фрагменты исключаются

### 6.2. Обрезка контекста
При переполнении лимита контекста Модели Б (>8192 токена) сначала удаляются избыточные сниппеты поиска. Запрос пользователя и факты Модели А оставляются без изменений, так как они являются наиболее важными для формирования ответа.

### 6.3. Кэширование ответов
Полезные результаты могут сохраняться в базу данных (Redis/SQLite) с ttl-временем 3600 секунд. При поступлении идентичного запроса система возвращает закэшированный ответ без повторной генерации.

## 7. ПРОМТ-Шаблон для модулей

Промты должны следовать формату ChatML с обязательными маркерами начала и конца секций. Изменение структуры токенов недопустимо и снижает качество генерации на 15–20%.

Модуль извлечения фактов EVA. Отвечай ТОЛЬКО фактами, максимум 3 предложения. Никаких рассуждений, вступлений или выводов. Только сухие данные. Язык ответа — Русский. Не используй Markdown.

Системный промт GGUF уточнения

Модуль анализа и расширения контекста EVA. Получаешь факты от Модуля А и результаты веб-поиска. 
Задача: сверь факты с поиском и укажи противоречия; расширь ответ концепциями из поиска; оцени этическую корректность без излишней морализации; сформируй развёрнутый ответ 5–7 предложений; используй ссылки на источники в формате [Источник N]. 
Приоритет тем запроса, язык — Русский. При противоречии приоритет у данных поиска.
