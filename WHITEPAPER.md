## 1. Теоретические основы

### 1.1. Проблема

Современные языковые модели основаны на механизме внимания (transformer):
- **Вычислительная сложность:** O(L²·D) для self-attention, где L — длина последовательности
- **Память контекста:** KV-cache растёт как O(L·D), достигая гигабайтов при длинных контекстах
- **Аппаратные требования:** десятки гигабайтов VRAM для инференса на длинных текстах

Эти ограничения делают трансформеры непрактичными для:
- Edge-устройств (смартфоны, NPU, робототехника)
- Длинных контекстов (>100K токенов)
- Реального времени с минимальной задержкой

**λ_d** (lambda-d) решает эту проблему, заменяя механизм внимания на **контент-зависимый спектральный оператор** с линейной сложностью O(L·D) и памятью O(D) (3.5 KB при D=896).

### 1.2. Рекурренция λ_d

**Основная идея:** вместо фиксированного линейного преобразования (как в SSM/Mamba) или попарного взаимодействия токенов (как в трансформерах), λ_d вычисляет для каждого токена свой линейный оператор A(h):

```
h_t = h_{t-1} + V · diag(α(h_t) ★ λ) · V^T · norm(h_{t-1} + conv([h_{t-1}, ..., h_{t-4}]))
```

где:
- **h_t ∈ ℝ^D** — скрытое состояние в момент t
- **V ∈ O(D) ⊂ ℝ^{D×D}** — ортогональная матрица (фиксированный собственный базис)
- **λ ∈ ℝ^K** — корни Фибоначчи (фиксированный спектр)
- **α(h) ∈ Δ^K** — контент-зависимые гейты (единственные обучаемые параметры в операторе)
- **conv** — depthwise causal 1D convolution (локальный контекст 4 токена)
- **norm** — RMS normalization

**Ключевое свойство:** оператор A(h) = V · diag(α(h) ★ λ) · V^T меняется от токена к токену, но его изменение контролируется всего K=4 скалярными гейтами. Это обеспечивает:
1. **Content-dependent routing** — разные токены по-разному активируют спектральные моды
2. **Линейную сложность** — O(D²) на токен (сравнимо с MLP), не O(L²)
3. **Память O(D)** — всё прошлое сжато в h_t размера D

### 1.3. Спектр Фибоначчи (λ_k)

**Определение:** λ_k — наибольший действительный корень полинома:

```
x^k = x^{k-1} + x^{k-2} + ... + x + 1       (k ≥ 2)
```

**Вывод:** Для k=2: x² = x + 1 → λ₂ = φ ≈ 1.618 (золотое сечение).
Для k=3: x³ = x² + x + 1 → λ₃ ≈ 1.839.
Предел при k → ∞: λ_k → 2.

**Численные значения:**

| k | λ_k | Горизонт памяти | Характер |
|---|-----|-----------------|----------|
| 2 | 1.61803 | Короткий (φ^{-t}) | Быстрое затухание |
| 3 | 1.83929 | Средний | Умеренное |
| 4 | 1.92756 | Длинный | Медленное |
| 5 | 1.96595 | Очень длинный | Почти постоянное |
| 6 | 1.98358 | Квази-постоянный | ~0.5% затухания/шаг |
| 7 | 1.99196 | Постоянный | <0.3% затухания/шаг |
| 8 | 1.99603 | Почти единичный | <0.2% затухания/шаг |

**Интуиция:** Каждое λ_k соответствует своей временно́й константе. Моды с малым λ_k отвечают за локальные, быстро меняющиеся паттерны (синтаксис, морфология). Моды с λ_k ≈ 2 отвечают за глобальный контекст (тема, дискурс).

**Вычислительный метод (Newton):**

```python
def fibonacci_roots(max_k=7):
    roots = []
    for k in range(2, max_k + 1):
        lo, hi = 1.0, 2.0
        for _ in range(100):
            mid = (lo + hi) / 2
            powers = mid ** torch.arange(k, -1, -1)
            f = powers[0] - powers[1:].sum()
            if f > 0: hi = mid
            else: lo = mid
        roots.append((lo + hi) / 2)
    return roots
```

Сходится за ~40 итераций бинпоиска (точность 2^{-40} ≈ 10^{-12}).

### 1.4. Ортогональный собственный базис V

V ∈ O(D) — случайная ортогональная матрица. Построение через произведение отражений Хаусхолдера:

```python
def random_orthogonal(D, n_reflections=32):
    V = eye(D)
    for _ in range(n_reflections):
        u = randn(D) / norm(randn(D))
        V = V - 2 * outer(V @ u, u)
    return V
```

**Математическое обоснование:**
- Каждое отражение Хаусхолдера H_u = I - 2·uu^T — ортогональная матрица (H_u^T·H_u = I)
- Произведение ортогональных матриц ортогонально
- 32 отражения достаточно для эргодического покрытия O(D)

**Зачем V?**
- Без V матрица diag(λ_eff) действует независимо по осям
- V смешивает оси, распределяя спектральное сжатие/растяжение по всем D измерениям
- V^T = V^{-1} — обращение за O(D²) без вычислений (просто транспонирование)
- Каждый слой имеет свой V_l (разные случайные ортогональные матрицы)

**Формальное определение:**

V разбита на K блоков размера M = D/K:

```
V = [V_1 | V_2 | ... | V_K] ∈ ℝ^{D×D}
V_k ∈ ℝ^{D×M}, V_i^T · V_j = I_{i=j} · I_M
```

Каждый блок V_k задаёт подпространство размерности M, которое масштабируется своим λ_k·α_k.

### 1.5. Контент-зависимое управление (gating)

Гейты α(h) — softmax-распределение весов между K корнями:

```
α(h) = softmax(4.0 · (W_gate · rms_norm(h) + b_gate))

где:
  W_gate ∈ ℝ^{D×K} — 3.6K параметров на слой
  b_gate ∈ ℝ^K      — 4 параметра на слой
  scale = 4.0        — температурный фактор
```

**Почему scale=4.0?** Без масштабирования (scale=1.0) softmax даёт почти равномерное распределение (энтропия ~1.38 из максимума ln(4) ≈ 1.386). При scale=4.0 гейты дифференцируются (энтропия ~0.15-0.38), что соответствует различной активации мод для разных токенов.

**Почему D×K, а не D×D?** Если бы каждый из D каналов имел свой гейт, было бы D²=800K параметров — на порядок больше. D×K=3584 — минимальная проекция, достаточная для содержательного routing.

### 1.6. Полная формула шага λ_d

```
Вход:  h ∈ ℝ^{B×L×D}
Шаг 1: h_conv = causal_conv1d(h)                  # (B, L, D)
Шаг 2: h_norm = rms_norm(h + h_conv)               # (B, L, D)
Шаг 3: gate_logits = h_norm @ W_gate + b_gate      # (B, L, K)
Шаг 4: α = softmax(4.0 · gate_logits)               # (B, L, K)
Шаг 5: λ_alpha = λ ★ α                              # (B, L, K)
Шаг 6: λ_eff = repeat_interleave(λ_alpha, M)        # (B, L, D)
Шаг 7: h_proj = h_norm @ V^T                        # (B, L, D)
Шаг 8: h_scaled = h_proj * λ_eff                    # (B, L, D)
Шаг 9: Δ = h_scaled @ V                             # (B, L, D)
Шаг 10: h_out = h + Δ                               # (B, L, D)
```

**No clamping:** норма Δ ограничена ||Δ|| ≤ max(λ)·√D ≈ 2·√D ≈ 60 — автоматически, без явного clamping.

## 2. Архитектура компонентов

### 2.1. Общая схема

```
tokens (B, L)
  ↓
Embedding (V×D) ─────────────── 44.8M params
  ↓ h₀
─── for l = 1..12: ──────────────────────────────
  ↓
  CausalConv1d (kernel=4, frozen) ─── 4.5K×12 = 54K
  ↓
  RMSNorm (learned weight) ──────── 896×12 = 10.8K
  ↓
  LDBlock:
    ↓ h @ W_gate + b_gate → softmax → α
    ↓ V · diag(α★λ) · V^T · h_norm
    ↓ h + Δ
  ↓
  BottleneckMLP (896→256→896, SiLU) ─── 459K×12 = 5.5M
  ↓
─── end for ──────────────────────────────────
  ↓
RMSNorm
  ↓
lm_head @ h → logits (V) ─────────── 44.8M
```

### 2.2. Embedding (слой токен-векторов)

```
Embedding: tokens ∈ [0, V) → h ∈ ℝ^D
Параметры: V × D = 50000 × 896 = 44.8M
```

Стандартное табличное представление. Каждый токен из словаря размера V=50K проецируется в D=896-мерное скрытое пространство.

**Замечание:** lm_head может быть связан с embedding (weight tying) — тогда параметры lm_head (44.8M) не добавляются. В текущей реализации они раздельные, что ухудшает обобщение, но стандартно для Phase 2.

### 2.3. CausalConv1d

```
CausalConv1d: (B, L, D) → (B, L, D)
Параметры: D × kernel × 1 = 896 × 4 × 1 = 3.6K (frozen)

F: conv1d(
    padding_left = kernel-1,
    groups = D,            # depthwise
    weight = D × 1 × 4,   # отдельный фильтр на канал
    bias = D
  )
```

**Depthwise 1D convolution** — каждый из D каналов свёртывается со своим фильтром длины 4. Фильтры фиксированы (инициализированы случайно, не обучаются).

**Causal padding:** padding слева на kernel-1 = 3 нуля, чтобы токен t не видел токен t+1 (причинность).

**Зачем conv?**

Без conv модель эквивалентна bag-of-words: она не отличает порядок слов. Causal conv даёт локальный контекст 4 токена, решая позиционную слепоту.

Критически важно: conv — единственный источник кросс-токенной информации в λ_d (кроме самого λ_d рекурренции через h).

### 2.4. RMSNorm

```
RMSNorm: x → x / RMS(x) · weight

RMS(x) = sqrt(mean(x_i²) + ε)
Параметры: weight ∈ ℝ^D (обучаемый, 896)
```

Нормализация по корню из среднего квадрата, не по стандартному отклонению (как LayerNorm). Разница в отсутствии вычитания среднего — RMSNorm сохраняет DC-компоненту.

### 2.5. LDBlock (ядро архитектуры)

```
LDBlock:
  Вход: h ∈ ℝ^{B×L×D}
  1. h_conv = CausalConv1d(h)
  2. h_norm = RMSNorm(h + h_conv, weight)
  3. gate_logits = (h_norm @ W_gate + b_gate) * 4.0
  4. α = softmax(gate_logits)                    # (B, L, K)
  5. λ_alpha = λ_k * α                           # (B, L, K)
  6. λ_eff = repeat_interleave(λ_alpha, M)       # (B, L, D), M = D//K
  7. h_proj = h_norm @ V^T                       # (B, L, D)
  8. h_scaled = h_proj * λ_eff                   # (B, L, D)
  9. Δ = h_scaled @ V                            # (B, L, D)
  10. return h + Δ
```

**Параметры LDBlock (один слой):**

| Компонент | Shape | Параметры | Обучаемые? |
|-----------|-------|-----------|------------|
| Conv weight | (896, 1, 4) | 3.6K | Нет |
| W_gate | (896, 4) | 3.6K | Да |
| b_gate | (4,) | 4 | Да |
| V (32 HH reflections) | (896, 896) | 803K | Нет |
| V_T (транспонированный V) | (896, 896) | 803K | Нет |
| λ_k | (4,) | 4 | Нет |
| RMS norm weight | (896,) | 896 | Да |
| **Итого** | | **~806K** | **~4.5K** |

**Ключевое:** из 806K параметров слоя только 4.5K (0.56%) — обучаемые контент-зависимые. Всё остальное — фиксированные геометрические структуры (V, λ, conv).

### 2.6. BottleneckMLP

```
BottleneckMLP:
  z = SiLU(W_up @ h_norm)          # D → bottleneck
  return W_down @ z                 # bottleneck → D

W_up:   (D, bottleneck)    = (896, 256)    = 229K
W_down: (bottleneck, D)    = (256, 896)    = 229K
Итого:  459K параметров на слой, полностью обучаемые
```

**Почему bottleneck 256, а не standard (4864)?**

В стандартных трансформерах MLP расширяет: D → 4D → D (896 → 3584 → 896, ~6.4M на слой). BottleneckMLP сжимает: D → bottleneck → D (896 → 256 → 896, 459K — в 14× меньше).

**Историческая справка:** В Phase 1 MLP был LoRA rank-16 поверх нулевой базы (только gates + LoRA обучались). Это давало эффективный ранг 16 для 4864 нейронов — 99.7% нейронов были линейно зависимы. Замена на dense bottleneck 256 дала полный ранг 256 при сопоставимом числе параметров.

### 2.7. ZeckendorfReadout (альтернативный выход)

**Проблема lm_head:** линейный слой D×V = 896×50000 = 44.8M — 47% всех параметров модели (при D=896). Он нужен только на последнем шаге, но его вес доминирует.

**Решение:** древовидный декодер на базе теоремы Цекендорфа.

**Теорема Цекендорфа (1939):** Каждое натуральное число N имеет единственное представление в виде суммы **НЕСОСЕДНИХ** чисел Фибоначчи:

```
N = Σ b_k · F_k, где b_k ∈ {0, 1}, b_k · b_{k+1} = 0
```

**Архитектура ZeckendorfReadout:**

```
Вход: h ∈ ℝ^D

Для k = K-1 до 0 (MSB to LSB):
  если state = 1:
    bit_k = 0 (принудительно — ограничение Цекендорфа)
  иначе:
    P(bit_k = 1 | h, state_k) = sigmoid(h · c_{k, state_k, 1})
  
log P(token | h) = Σ_k log P(bit_k | h, state_k)

Параметры: c_{k, state, bit} ∈ ℝ^D, K × 2 × 2 × D = 4·K·D
При K=20 (для V=50K): 4 × 20 × 896 = 71.7K (против 44.8M у lm_head)
```

**Преимущества:**
- 71.7K против 44.8M → в **625× меньше**
- Декодирование за K=20 шагов (дерево глубины 20 вместо матричного умножения V×D)
- Встроенный бинарный код — интерпретируемость

**Недостатки:**
- ~35% токенов vocab=50K не имеют Zeckendorf-представления (F_K - 1 < V_макс)
- Требует дообучения (дистилляция с lm_head)
- Последовательное декодирование (K шагов вместо одного)

### 2.8. LDStack (полная модель)

```
LDStack = [LDBlock + BottleneckMLP] × L слоёв с residual + final RMSNorm
```

```
for l in range(L):
    h = LDBlock[l](h)                    # спектральное преобразование
    h_norm = RMSNorm(h, final_norm_w)    # pre-MLP norm
    h = h + BottleneckMLP[l](h_norm)     # нелинейная проекция

return RMSNorm(h, final_norm_w)          # финальная нормализация
```

Каждый LDBlock использует свой V_l, свой W_gate_l, свои λ_k. BottleneckMLP — единственный источник нелинейности (SiLU) и единственный слой, где D-мерное пространство проецируется через узкое горлышко (256).

## 3. Математические механизмы

### 3.1. Как работает λ_d: геометрическая интуиция

**Три ключевые идеи:**

1. **Спектральное разложение:** h_norm проецируется на ортогональный базис V. В этом базисе каждая координата масштабируется своим коэффициентом λ_eff_k ∈ [1.6, 2.0] (или [−2.0, −1.6] при α_k → 0 — но softmax гарантирует α_k > 0).

2. **Блоковое масштабирование:** D координат разбиты на K=4 группы по M=224. Каждая группа масштабируется на λ_k · α_k. Это даёт K независимых «ручек» управления спектром.

3. **Content-dependent routing:** α_k = f(h) — гейты зависят от содержимого h через W_gate. Это единственный контент-зависимый элемент.

**Сравнение с классическими архитектурами:**

| Архитектура | Оператор | Зависимость от контента | Сложность |
|-------------|----------|------------------------|-----------|
| LSTM | σ(W·x + U·h) | Да (гейты) | O(D²) |
| Transformer | softmax(QK^T)V | Да (attention) | O(L²) |
| Mamba/SSM | A·h + B·x + C·h | Нет (фикс. A) | O(L) |
| **λ_d** | V·diag(α·λ)·V^T | **Да (гейты α)** | **O(D²)** |

### 3.2. Почему V frozen + learned gates = работает (но spectral gates = нет)

Это ключевое открытие Phase 2, подтверждённое экспериментально:

**Эксперимент:** Замена learned gates (W_gate·h + b) на spectral energy gates (||V_k^T·h||² / Σ||V_j^T·h||²).

**Результат:**

| Gates | PPL | Entropy α |
|-------|-----|-----------|
| Learned (W_gate) | **125.5** | 0.153 |
| Spectral L2 (energy) | 413.0 | 1.383 |
| Spectral L∞ (τ=5) | 3967.7 | 0.104 |

**Почему spectral gates провалились?**

**Анализ V-энергии до и после forward pass:**

| Слой | V-энергия по блокам [0, 1, 2, 3] | Энтропия |
|------|----------------------------------|----------|
| Embedding (вход) | [222, 224, 223, 225] | 1.383 |
| После 12 слоёв | [7571, 12440, 9272, 10554] | **1.083** |

**Объяснение по шагам:**

1. **На входе (embedding):** h изотропен в V-пространстве. Все 4 блока имеют одинаковую энергию (~224). Причина: V — случайная ортогональная матрица, не связанная с данными.

2. **Learned gates** (W_gate·h + b): W_gate ∈ ℝ^{896×4} — это 4 проекции h из 896-мерного пространства в 4 скаляра. Даже если h изотропен в V-базисе, он сохраняет всю свою 896-мерную структуру в исходном базисе. W_gate улавливает эту структуру и вычисляет содержательный α.

3. **Spectral energy gates** (||V_k^T·h||²): α_k зависит только от проекции h на V-подпространство. Поскольку V-подпространства случайны, энергия h в них не несёт содержательной информации — только шум.

4. **Bootstrapping-цикл (learned gates):**

```
Слой 0: h (изотропный) → W_gate(h) → α₀ → V·diag(α★λ)·V^T → h₁ (слабо анизотропный)
Слой 1: h₁ (анизотропный) → W_gate(h₁) → α₁ (более селективный) → h₂ (сильно анизотропный)
...
Слой 12: h₁₂ → V-энергия [7571, 12440, 9272, 10554] — в 1.64× разница между блоками
```

5. **Коллапс spectral gates:** α_k = ||V_k^T·h||² / Σ||V_j^T·h||² ≈ 0.25 для всех k (изотропия). Все 4 блока получают одинаковое усиление λ_k·0.25 ≈ 0.5. Разница между λ_k (1.618 → 0.405, 1.928 → 0.482) мала. Оператор близок к identity — модель теряет routing.

**Математическая формализация:**

Для случайного ортогонального V с блоками размера M = D/K:

```
Для изотропного h ∈ ℝ^D:
  E[||V_k^T·h||²] = (M/D) · ||h||² = ||h||² / K    (одинаково для всех k)
  Var[||V_k^T·h||²] = 2M²/D² · ||h||⁴ ≈ 2/K² · ||h||⁴
```

Отношение сигнал/шум: E/σ = 1/√(2M) ≈ 0.047 для M=224 — шум доминирует над сигналом.

**Вывод:** 43K параметров W_gate — это не «лишние 0.04%» модели. Это **единственный механизм**, связывающий содержимое h с выбором спектральных мод. Без него V — просто случайный базис, не несущий информации о данных.

### 3.3. Эквивалентность attention head specialization

**Удивительный факт:** learned gates в λ_d и attention heads в трансформерах решают одну и ту же задачу — routing информации на основе содержимого.

| λ_d | Transformer |
|-----|-------------|
| V_k — подпространство базиса (фикс.) | W_value_k — value проекция (обучаемая) |
| α_k = softmax(W_gate·h) — гейт | A_ij = softmax(Q_i·K_j/√d) — attention |
| λ_k — временна́я константа | Positional encoding (встроенная) |
| λ_d масштабирует V-блок | Attention-weighted sum values |
| **43K** параметров гейтов | **~50M** параметров Q,K,V (LLaMA-7B) |

**Bootstrapping идентичен:**

```
Transformer слои:
  h → MultiHeadAttention(h) → h' (анизотропные head-паттерны)
  → FFN(h') → h'' → следующий слой видит структуру → глубже
```

В трансформерах attention heads через обучение специализируются: одни на синтаксисе, другие на семантике, третьи на позиционных паттернах. Это та же анизотропия, что и V-энергия в λ_d, но в пространстве attention patterns.

### 3.4. Почему V обязан быть frozen (пока)

1. **V задаёт ортогональный базис.** Произведение 32 Householder отражений гарантирует V^T·V = I. Если V — Parameter (обучаемый), ортогональность не сохранится.

2. **Нарушение ортогональности** означает:
   - Потерю V^T = V^{-1} (обращение становится нетривиальным)
   - Возможный коллапс (все векторы V_k в одном направлении)
   - Нестабильность норм ||V·diag(λ)·V^T·h|| (может расти неограниченно)

3. **Восстановление ортогональности** требует:
   - Procrustes projection после каждого шага: V ← V · (V^T·V)^{-1/2} (дорого, O(D³))
   - Или Riemannian optimization на многообразии Stiefel (сложно, нестабильно)
   - Или Cayley transform: V ← (I + S/2)·(I - S/2)^{-1}·V (экспоненциальный map)

**Для Phase 3** планируется сделать V обучаемым с Procrustes projection каждые N шагов.

### 3.5. Ансамбль λ-мод

Стек из 12 LDBlock с разными V_l создаёт ансамбль независимых λ_d операторов:

```
h_out = h_0 + Σ_l Δ_l
Δ_l = V_l · diag(α_l(h) ★ λ) · V_l^T · norm(h_l + conv(h_l))
```

Каждый слой:
- Имеет свой случайный V_l (независимый)
- Вычисляет свой α_l(h) (разные специализации)
- Видит разный h на входе (после предыдущих слоёв)

**Результат:** 12 независимых «экспертов», каждый со своим спектральным профилем. Гейты разных слоёв действительно специализируются (подтверждено экспериментально):

```
Слой 0: α = [0.282, 0.270, 0.329, 0.119]  H = 0.203 — почти равномерный
Слой 5: α = [0.094, 0.152, 0.096, 0.658]  H = 0.139 — доминирует мода 3
Слой 7: α = [0.324, 0.239, 0.155, 0.281]  H = 0.382 — самая равномерная
Слой 8: α = [0.290, 0.240, 0.195, 0.275]  H = 0.304 — смешанная
```

Средняя энтропия по слоям: 0.207 (максимум ln(4) = 1.386). Гейты используют ~15% доступной ёмкости, сильно селективны.

### 4.2. Оптимизация

**Оптимизатор:** AdamW (β₁=0.9, β₂=0.95, eps=1e-8, weight_decay=0.01)

**LR schedule:**
```
Linear warmup 5% → Cosine decay:
  step < warmup_steps: lr = max_lr × step / warmup_steps
  step ≥ warmup_steps: lr = max_lr × 0.5 × (1 + cos(π × (step - warmup) / (total - warmup)))
  
  max_lr = 1e-3
  warmup_steps = total_steps × 0.05
```

**Gradient clipping:** 1.0 (global norm)

**Loss:** Cross-entropy (token-level)

### 4.3. Gradient Accumulation

Физический batch = 4 (лимит VRAM MX550: 2GB). Для имитации batch=32:

```
loss = cross_entropy(logits, targets) / ACCUM_STEPS  # 8
loss.backward()
if step % ACCUM_STEPS == 0:
    clip_grad_norm(1.0)
    optimizer.step()
    optimizer.zero_grad()
```

Градиенты накапливаются 8 микрошагов, затем применяются. Это даёт:
- Более стабильные градиенты (меньше шума)
- Лучшая оценка batch statistics
- Возможность train на MX550 2GB

### 4.4. Чекпоинты и resume

```
save_checkpoint: model_state, optimizer_state, step, epoch, best_ppl, config
find_latest_ckpt: сортировка glob('checkpoints/phase2_*.pt') по времени
resume: загрузка последнего чекпоинта → восстановление step, epoch, optimizer
```

**Авто-resume** при повторном запуске: скрипт находит последний чекпоинт и продолжает с того же шага, восстанавливая все состояния.

### 4.5. Precision

| Среда | Precision | Причина |
|-------|-----------|---------|
| MX550 (local) | fp32 | fp16 даёт NaN на 2GB карте |
| T4 (Colab) | fp16 (AMP) | 16GB VRAM, стабильно |
| Checkpoint storage | fp16 (weight-only) | 229MB/ckpt (vs 400MB fp32) |

Colab чекпоинты сохраняются в fp16 для экономии места на Google Drive (8GB free):

```python
ckpt = {
    'model_fp16': {k: v.half() for k, v in model.state_dict().items()},
    'step': step, 'loss': loss,
    'batch_size': BATCH_SIZE,  # для rescale шагов
}
```

### 4.6. Batch size scaling на Colab

Из-за ограничений T4 (16GB) batch пришлось увеличивать: 8 → 32 → 64. Каждое изменение требует rescale шагов:

```
step_new = step_old × batch_old / batch_new
```

Зачем? LR scheduler считает прогресс в шагах. Если batch удвоился, один шаг видит вдвое больше данных. Rescale выравнивает.

### 7.5. Серверное размещение: скорость инференса решает

На сервере 10 GB лишней памяти не проблема. Проблема — **скорость инференса на длинном контексте**.

Трансформер на 1M токенов:
- Prefill: O(L²) attention → **30-60 сек** на A100 (8B модель)
- Каждый новый токен: O(L·D) attention → **замедляется линейно** с длиной
- KV-cache: 16 GB (8B, L=1M) — влазит, но prefill чудовищно медленный

λ_d hierarchy:
- Prefill: O(L·D) → **0.3-1 сек** на A100
- Каждый новый токен: O(D²) → **константа**, не зависит от длины
- KV-cache: 28 KB — не влияет ни на что

**Сравнение FLOPs на токен (8B модель, D=4096):**

| Операция | Transformer | λ_d hierarchy | λ_d быстрее в |

| MLP | 8·D² = 134M (SwiGLU) | 0.5·D² = 8.4M (bottleneck) | **16×** |
| V проекция | — | ≈ 1·D² = 16.8M (биполярная: ±1, без умножений) | **бесплатно** |
| **На слой** | **~200M** | **~25M** | **8×** |
| **На токен (L=48)** | **~9.6B** | **~1.2B** | **8×** |
| **Иерархия (страйд)** | 9.6B (каждый токен) | Level 0: 16M (каждый) + Level 1: 1.3M (1/64) + Level 2: 0.16M (1/2048) = **~17.5M** | **~550×** |

## 8. Реальная длина контекста λ_d

### 8.1. Механизм памяти

В трансформерах контекст хранится в KV-cache — матрице размера L×D для каждого слоя, где каждый токен занимает свою строку. Доступ к токену i осуществляется через attention score A_{t,i}. Длина контекста = L физически (столько строк), а ограничение — только VRAM.

В λ_d контекст хранится **в скрытом состоянии h ∈ ℝ^D** — один вектор на слой, не зависящий от длины:

```
h_t = h_{t-1} + Δ_t
Δ_t = V · diag(α★λ) · V^T · rms_norm(h_{t-1} + conv(h_{t-1}, ..., h_{t-4}))
```

После T шагов: h_T = h_0 + Σ_{t=1}^{T} Δ_t. Состояние — **сумма всех прошлых обновлений**. Старые токены НЕ затухают. Вопрос в том, как долго информация о конкретном токене остаётся извлекаемой из h.

### 8.2. Анализ ёмкости

**Абсолютный предел (линейная ёмкость):**

h ∈ ℝ^D может хранить максимум D независимых скаляров. Каждый токен вносит Δ ∈ ℝ^D — один вектор. После D шагов векторы Δ_i становятся линейно зависимы. **Теоретический максимум:** D = **896 токенов**.

**Ограничение RMSNorm (только направление):**

RMSNorm делит h на RMS(h), фиксируя амплитуду. Это превращает пространство состояний из ℝ^D в сферу S^{D-1}. Информация хранится ТОЛЬКО в направлении h. На сфере максимум D-1 = **895 степеней свободы**.

**Ограничение α-routing (блоки, не каналы):**

V разбит на K=4 блока по M=224. Каждый токен вносит K=4 числа (α_k) — routing-информацию. За t шагов вносится t·K скаляров. После D/K = 224 шагов каждый V-блок покрыт проекциями от разных токенов. **Практический предел до интерференции:** 224 токена.

**Затухание vs перезапись:**

```
В SSM/Mamba: h_t = λ·h_{t-1} + B·x_t, λ < 1 → экспоненциальное затухание
В λ_d:       h_t = h_{t-1} + V·diag(α★λ)·V^T·..., λ > 1 → экспансия, не затухание
```

В λ_d старые токены **не затухают** — они перезаписываются новыми через проекции на V-подпространства. Δ_i содержит проекцию h_{i-1} на V-подпространства, взвешенную α. Поскольку h_{i-1} содержит все предыдущие Δ, каждый новый шаг перезаписывает часть старых компонент.

**Время «полужизни» старой информации:**

При среднем λ·α ≈ 1.82 и K=4 блоках, каждый новый шаг вносит 4·M = 896 чисел (полный D-вектор). Вклад старого токена на шаге t+i уменьшается как:

```
old_contrib ∝ Π_{j=1}^{i} (λ_k · α_k^{(j)})
```

Поскольку λ_k · α_k > 1 (среднее 1.82), вклад старого токена **растёт** через последующие шаги, но распределяется по новым направлениям. Это не затухание — это диффузия.

### 8.3. Эмпирическая валидация

**Метод:** forward pass на непрерывных последовательностях длиной до 256 токенов (2 склеенных чанка из обучения), измерение PPL по позициям.

**Результат (модель 105M, D=896, frozen V):**

```

PPL на позиции 8:           78.5
PPL на позиции 32–120:      80–130 (стабильно, ~100)

```

### 8.4. Итоговая оценка

| Уровень | Длина контекста | Обоснование |
|---------|----------------|-------------|
| **Надёжный (train distribution)** | **128 токенов** | Модель обучена на L=128, PPL стабилен на всём протяжении |
| **Экстраполяция (эмпирическая)** | **256–512 токенов** | Динамика не меняется при L > 128; conv + λ_d архитектура length-agnostic |

| **Теоретический максимум (сжатие)** | **~4480 токенов** | При ~2 бит/токен сжатой информации в 896×10 бит |

**Ключевое преимущество:** в отличие от трансформеров, где контекст линейно расширяет KV-cache (O(L·D) памяти), λ_d хранит весь контекст в одном векторе h (O(D) памяти). Цена — интерференция (старые токены частично перезаписываются новыми), а не физический лимит памяти.

## 9. Phase 3: Иерархический λ_d (FCF-style)

### 9.1. Ограничения текущей архитектуры

Текущая λ_d (D=896, K=4, L=12) имеет фундаментальные ограничения:

1. **Один V на все моды.** K=4 блока недостаточно для богатой иерархии. Больше K → меньше M = D/K → слабее подпространства.

2. **Нет иерархии времён.** Все 12 слоёв используют одни и те же λ_k. Нет специализации слоёв на разных масштабах.

3. **Случайный V.** V — frozen случайная ортогональная матрица. Её блоки не специализированы под данные.

4. **Плоское residual.** h проходит через все слои последовательно. Нет skip-connections между разными масштабами.

### 9.2. Идея: FCF (Fibonacci Convolution Fields)

**Ключевая идея FCF:** множество независимых λ_d-клеток, каждая со своим V, λ, и входом. Клетки общаются через binding/unbinding операции.

```
Уровень k:
  state_k[t] = λ_k · state_k[t-1] + bind(vec_k[t], pos_shift[t])

где:
  λ_k — свой корень Фибоначчи на уровень
  state_k ∈ ℝ^{M} — состояние на уровне k (маленькое M=128-256)
  vec_k — вход с предыдущего уровня
  bind — VSA binding (покоординатное умножение с FFT)
```

**Иерархия:**

```
Уровень 1 (λ₁ ≈ 1.618): токены, морфология
    ↓ bind/unbind
Уровень 2 (λ₂ ≈ 1.839): синтаксис, фразы
    ↓ bind/unbind
Уровень 3 (λ₃ ≈ 1.928): семантика предложений
    ↓ bind/unbind
...
Уровень K (λ_K ≈ 2.0): тема документа
```

### 9.3. Предлагаемая архитектура Phase 3

```
Вход: tokens → Embed(D)

Уровень 0 (быстрый, D=1024, λ=1.618):
  state_0[t] = λ₁·state_0[t-1] + bind(V₀·embed[t], pos[t])
  output_0 = unbind(state_0, pos[t])
  
Уровень 1 (средний, D=1024, λ=1.839):
  state_1[t] = λ₂·state_1[t-1] + bind(V₁·output_0, pos[t])
  output_1 = unbind(state_1, pos[t])

...

Выход: Combine(output_0, ..., output_K) → RMSNorm → lm_head

Total params: ~1-8B в зависимости от K, D, глубины
```

**Сравнение с текущей λ_d (Phase 2):**

| Аспект | Phase 2 | Phase 3 (FCF) |
|--------|-------------------|---------------|
| V | 1 × 896 = 803K | K × 128-256 = 2-8M |
| Гейты | W_gate: 896×4 | VSA binding (0 params) |
| Масштабы | 4 (один слой) | K (независимые) |
| Коммуникация | Residual stream | Binding/unbinding |
| Параметры | 105M | 1-8B |

## 11. Приложение: код

### 10.1. LDBlock (ядро)

```python
class LDBlock(nn.Module):
    def __init__(self, cfg, layer_idx, lambda_roots):
        super().__init__()
        self.D = cfg.D
        self.K = cfg.n_modes
        self.block_size = cfg.D // cfg.n_modes
        self.conv = CausalConv1d(cfg.D, kernel_size=4)
        V_init = random_orthogonal(cfg.D, n_reflections=32)
        self.register_buffer('V', V_init)
        self.register_buffer('V_T', V_init.T.contiguous())
        self.W_gate = nn.Parameter(torch.randn(cfg.D, cfg.n_modes) * 0.01)
        self.b_gate = nn.Parameter(torch.randn(cfg.n_modes) * 0.01)
        self.register_buffer('lambda_k', lambda_roots[:cfg.n_modes])
        self.register_buffer('input_ln_w', torch.ones(cfg.D))

    def forward(self, h, return_gates=False, residual=True):
        B, L, D = h.shape
        h_conv = self.conv(h)
        h_norm = rms_norm(h + h_conv, self.input_ln_w)
        gate_logits = (h_norm @ self.W_gate) + self.b_gate
        gate_logits = gate_logits * 4.0
        alpha = F.softmax(gate_logits, dim=-1)
        lambda_alpha = self.lambda_k * alpha
        lambda_eff = lambda_alpha.repeat_interleave(self.block_size, dim=-1)
        h_proj = h_norm @ self.V_T
        h_scaled = h_proj * lambda_eff
        delta = h_scaled @ self.V_T.T
        h_out = h + delta if residual else delta
        return (h_out, alpha) if return_gates else h_out
```

### 10.2. CausalConv1d

```python
class CausalConv1d(nn.Module):
    def __init__(self, D, kernel_size=4):
        super().__init__()
        self.kernel_size = kernel_size
        w = torch.randn(D, 1, kernel_size) * 0.1
        self.register_buffer('weight', w)
        self.register_buffer('bias', torch.zeros(D))

    def forward(self, x):
        x_perm = x.transpose(1, 2)          # (B, L, D) → (B, D, L)
        pad = self.kernel_size - 1
        x_pad = F.pad(x_perm, (pad, 0))     # left-only padding (causal)
        out = F.conv1d(x_pad, self.weight, bias=self.bias,
                       groups=self.weight.shape[0])
        return out.transpose(1, 2)          # (B, D, L) → (B, L, D)
```

### 10.3. BottleneckMLP

```python
class BottleneckMLP(nn.Module):
    def __init__(self, D, bottleneck=256):
        super().__init__()
        self.up = nn.Linear(D, bottleneck, bias=False)
        self.down = nn.Linear(bottleneck, D, bias=False)

    def forward(self, x):
        return self.down(F.silu(self.up(x)))
```

### 10.4. RMSNorm

```python
def rms_norm(x, weight, eps=1e-6):
    rms = x.norm(dim=-1, keepdim=True) / (x.shape[-1] ** 0.5)
    rms = rms.clamp(min=eps)
    return x / rms * weight
```

### 10.5. Random Orthogonal (Householder)

```python
def random_orthogonal(D, n_reflections=32):
    V = torch.eye(D, dtype=torch.float32)
    for _ in range(n_reflections):
        u = torch.randn(D, dtype=torch.float32)
        u = u / (u.norm() + 1e-10)
        V = V - 2 * torch.outer(V @ u, u)
    return V
```

### 10.6. Fibonacci roots (Newton bisection)

```python
def fibonacci_roots(max_k=7):
    roots = []
    for k in range(2, max_k + 1):
        lo, hi = 1.0, 2.0
        for _ in range(100):
            mid = (lo + hi) / 2
            powers = mid ** torch.arange(k, -1, -1, dtype=torch.float64)
            f = powers[0] - powers[1:].sum()
            if f > 0: hi = mid
            else: lo = mid
        roots.append((lo + hi) / 2)
    return torch.tensor(roots, dtype=torch.float32)
```

### 10.7. LDStack (полная модель)

```python
class LDStack(nn.Module):
    def __init__(self, cfg):
        super().__init__()
        lambda_roots = fibonacci_roots(cfg.n_modes + 1)
        self.layers = nn.ModuleList([
            LDBlock(cfg, i, lambda_roots) for i in range(cfg.n_layers)
        ])
        self.mlps = nn.ModuleList([
            BottleneckMLP(cfg.D, cfg.bottleneck) for _ in range(cfg.n_layers)
        ])
        self.register_buffer('final_norm_w', torch.ones(cfg.D))

    def forward(self, h, return_gates=False):
        gates = [] if return_gates else None
        for lidx in range(self.n_layers):
            h, alpha = self.layers[lidx](h, return_gates=True)
            if return_gates: gates.append(alpha)
            h_norm = rms_norm(h, self.final_norm_w)
            h = h + self.mlps[lidx](h_norm)
        h_out = rms_norm(h, self.final_norm_w)
        return (h_out, torch.stack(gates)) if return_gates else h_out
```
