# K6: EventBus PriorityQueue работает как FIFO - Углублённый анализ

## Конкретное место проблемы

**Файл:** eva_ai/core/event_bus.py  
**Строки:** 19-24 (EventPriority), 275 (расчёт приоритета)

---

## 1. Описание проблемы

### 1.1 Инвертированный EventPriority Enum

`python
# Строка 19-24
class EventPriority(Enum):
    LOW = 1       # value = 1
    NORMAL = 2   # value = 2
    HIGH = 3     # value = 3
    CRITICAL = 4 # value = 4
`

**Проблема:** Enum определяет приоритеты так, что **меньшее число = выше приоритет** (LOW=1 означает высший приоритет). Это противоречит интуитивному пониманию, где CRITICAL должен быть больше чем LOW.

### 1.2 Инверсия при вычислении приоритета очереди

`python
# Строка 275
priority_value = 5 - event.priority.value
`

Эта формула **инвертирует** значение для PriorityQueue:

| Событие | enum.value | priority_value | Порядок в очереди |
|---------|------------|-----------------|-------------------|
| CRITICAL | 4 | 1 | 1-й (самый высокий) |
| HIGH | 3 | 2 | 2-й |
| NORMAL | 2 | 3 | 3-й |
| LOW | 1 | 4 | 4-й (самый низкий) |

**Работает правильно**, но запутанно. Формула 5 - value компенсирует перевёрнутый enum.

### 1.3 Несогласованность с event_system.py

В старой системе (event_system.py, строка 135):
`python
listeners.sort(key=lambda cb: getattr(cb, '_event_priority', priority_override), reverse=True)
`

Здесь 
everse=True означает: **БОЛЬШЕ число = ВЫШЕ приоритет**.

**Противоречие:**
- event_bus.py: меньше число в очереди = выше приоритет
- event_system.py: больше число = выше приоритет

## 2. Сравнительная таблица систем приоритетов

| Аспект | event_bus.py (новая) | event_system.py (старая) |
|--------|---------------------|---------------------------|
| **Направление** | Меньше = выше | Больше = выше |
| **Диапазон подписчика** | 0-255 (документация), 5 (код) | 0-10 |
| **Диапазон события** | LOW=1, CRITICAL=4 | system_health_check=10, memory_optimization=-3 |
| **Механизм сортировки** | PriorityQueue (автоматически) | list.sort(reverse=True) |
| **Формула** | 5 - event.priority.value | getattr(cb, '_event_priority', 5) |

## 3. Влияние на систему

### 3.1 Текущее влияние (минимальное)
- Система **функционирует корректно** - CRITICAL события обрабатываются раньше LOW
- Путаница для разработчиков при отладке приоритетов

### 3.2 Потенциальные проблемы
1. **Ошибки при расширении системы**: разработчик может ожидать CRITICAL > HIGH > NORMAL > LOW
2. **Несогласованность интерфейсов**: компоненты могут по-разному интерпретировать приоритеты
3. **Сложность рефакторинга**: инверсия 5 - value затрудняет изменения

### 3.3 Непоследовательность в docstring
`python
# Строка 159
priority: Приоритет обработчика (0=высший, 255=низший, по умолчанию 128)
`
**Реальность:**
`python
# Строка 152
priority: int = 5
`

Документация обещает диапазон 0-255 с дефолтом 128, а код использует 0-255 с дефолтом 5.

## 4. Варианты исправления

### Вариант 1: Исправить EventPriority Enum (РЕКОМЕНДУЕТСЯ)

**Принцип:** Сделать так, чтобы PriorityQueue работала напрямик без инверсии.

`python
class EventPriority(Enum):
    CRITICAL = 1  # Самый высокий приоритет = 1
    HIGH = 2
    NORMAL = 3
    LOW = 4       # Самый низкий приоритет = 4
`

Тогда:
- PriorityQueue.get() возвращает элемент с **наименьшим** priority_value
- CRITICAL=1 -> обрабатывается первым
- LOW=4 -> обрабатывается последним
- Формула: priority_value = event.priority.value (БЕЗ ИНВЕРСИИ!)

### Вариант 2: Оставить инверсию, исправить документацию

**Минимальные изменения:**
1. Исправить docstring в subscribe() (строка 159)
2. Добавить комментарий о принципе работы PriorityQueue

## 5. Рекомендация

**Рекомендуется Вариант 1 (исправление EventPriority):**

1. Изменить порядок значений в EventPriority enum:
`python
class EventPriority(Enum):
    CRITICAL = 1
    HIGH = 2
    NORMAL = 3
    LOW = 4
`

2. Убрать инверсию в строке 275:
`python
# Было:
priority_value = 5 - event.priority.value
# Стало:
priority_value = event.priority.value
`

**Преимущества:**
- Код становится само-документируемым
- Принцип меньше = выше работает напрямую без магических формул
- Соответствует интуитивному пониманию приоритетов
