# Системный Поток EVA-Ai (System Flow)

**Версия:** 3.1  
**Дата:** 2026-04-12  
**Архитектура:** Pie UnifiedGenerator с трёхэтапной итеративной генерацией

---

## 1. Обзор Системного Потока

Система EVA-Ai построена на принципах когнитивной обработки информации. Каждый запрос проходит через несколько этапов обработки, прежде чем будет сформирован финальный ответ. Ключевой особенностью является трёхэтапная итеративная генерация, где результаты каждого этапа становятся входными данными для следующего.

### 1.1 Высокоуровневая архитектура потока

Система состоит из следующих основных компонентов, соединённых в единый поток данных:

**Web-интерфейс** является точкой входа для пользовательских запросов. Он обеспечивает визуализацию чата, отображение рассуждений системы и предоставляет возможность загрузки файлов. При получении сообщения от пользователя интерфейс формирует запрос и передаёт его на обработку.

**CoreBrain** выступает центральным координатором всей системы. Он управляет жизненным циклом компонентов, обрабатывает запросы через специализированные модули и обеспечивает взаимодействие между различными частями системы. CoreBrain инициализирует все необходимые компоненты при запуске и координирует их работу во время обработки запросов.

**QueryProcessor** отвечает за предварительную обработку запросов. Он выполняет анализ запроса, определяет необходимость веб-поиска, проверяет этические ограничения и решает, какой путь обработки выбрать. Этот компонент также управляет кэшированием результатов для повышения производительности.

**UnifiedGenerator** является ядром генерации текста. Он реализует трёхэтапную итеративную архитектуру, где три специализированные модели работают последовательно для создания качественного ответа. Этот компонент обеспечивает выбор правильной модели на основе типа запроса и координацию между этапами генерации.

**FractalGraphV2** представляет собой систему долговременной памяти. Она хранит историю диалогов, извлечённые концепты и их связи, а также противоречия, обнаруженные в процессе работы. При генерации ответа система обращается к графу для получения релевантного контекста.

## 2. Детальный Поток Обработки Запроса

### 2.1 Этап приёма и первичной обработки

Когда пользователь отправляет сообщение через веб-интерфейс, система выполняет ряд первичных проверок и преобразований. На этом этапе проверяется аутентичность пользователя через систему сессий, анализируется содержимое сообщения на наличие запрещённого контента через EthicsFramework, и оценивается необходимость веб-поиска для дополнения ответа актуальной информацией.

Система сессий отслеживает активные диалоги и поддерживает контекст между сообщениями. Каждый пользователь имеет уникальный идентификатор сессии, который используется для загрузки соответствующей истории. Это позволяет системе учитывать предыдущие сообщения при формировании ответа.

EthicsFramework выполняет критически важную функцию фильтрации контента. Он анализирует каждое входящее сообщение на соответствие этическим нормам, определяет уровень риска и при необходимости блокирует недопустимые запросы. Этот компонент также генерирует предупреждения для контента сомнительного характера, но не блокирует его полностью.

Анализ необходимости веб-поиска определяет, требуется ли актуальная информация из интернета. Система использует эвристики для определения запросов, требующих свежих данных, таких как вопросы о текущих событиях, технические запросы с актуальными требованиями или запросы о малоизвестных темах.

### 2.2 Этап маршрутизации и выбора стратегии

После первичной обработки система определяет оптимальный путь обработки запроса. Этот этап критически важен для обеспечения эффективности и качества ответа.

Для простых приветствий и коротких запросов система использует быстрый кэш ответов. Этот механизм хранит заранее подготовленные ответы на типичные приветствия, что позволяет мгновенно реагировать без затраты ресурсов на генерацию. Проверка кэша занимает миллисекунды и существенно улучшает воспринимаемую скорость отклика.

Если запрос не найден в быстром кэше, система проверяет общий кэш токенов. Гибридная система кэширования хранит как горячие (в памяти GPU), так и холодные (на диске) кэши. Это позволяет повторно использовать результаты генерации для аналогичных запросов, экономя вычислительные ресурсы.

При отсутствии кэшированного результата система переходит к полному циклу генерации. На этом этапе определяется, какой генератор использовать — UnifiedGenerator с трёхэтапной итеративной генерацией или один из резервных механизмов. Выбор зависит от конфигурации системы и доступности моделей.

### 2.3 Этап трёхэтапной итеративной генерации

Центральным элементом обработки запроса является UnifiedGenerator, реализующий трёхэтапную итеративную архитектуру генерации. Этот подход обеспечивает высокое качество ответов через последовательное уточнение.

**Первый этап — LOGIC (первичный ответ)** использует модель LOGIC для генерации краткого, структурированного ответа на запрос. На этом этапе модель анализирует запрос, извлекает ключевые факты и формирует базовую структуру ответа. Результат первого этапа представляет собой сжатый ответ, охватывающий основные аспекты запроса.

Первый этап выполняет несколько функций. Он определяет ключевые термины и концепты, которые будут развиваться в последующих этапах. Модель также устанавливает логическую структуру ответа, которая послужит основой для расширения. Важно отметить, что на этом этапе система извлекает релевантные концепты из FractalGraphV2 для включения в контекст.

**Второй этап — CONTEXT (расширение)** получает результат первого этапа и расширяет его с помощью модели CONTEXT. Эта модель имеет больший контекстный размер и способна обрабатывать более длинные последовательности. Она использует результат LOGIC как отправную точку и развивает каждый упомянутый концепт.

На втором этапе происходит несколько важных процессов. Модель получает дополнительный контекст из FractalGraphV2, включающий релевантные концепты и историю предыдущих диалогов. Расширенный контекст позволяет добавить детали, примеры и объяснения, которые отсутствовали в кратком ответе. Результат второго этапа представляет собой развёрнутый текст с развитыми идеями.

**Третий этап — LOGIC (рефлексия и коррекция)** является ключевым инновационным элементом архитектуры. На этом этапе модель LOGIC получает результаты первых двух этапов и выполняет рефлексивный анализ.

Рефлексия включает несколько аспектов. Система проверяет согласованность между кратким ответом первого этапа и расширенным контекстом второго этапа. Если обнаружены противоречия, модель корректирует финальный ответ для их устранения. Также система извлекает ключевые концепты и идеи из расширенного контекста, показывая как они связаны с запросом пользователя. Наконец, модель формирует финальный, согласованный ответ учитывая всю информацию.

Результат третьего этапа представляет собой финальный ответ, прошедший проверку на согласованность и обогащённый ключевыми концептами. Этот ответ возвращается пользователю и сохраняется в историю диалогов и FractalGraphV2.

### 2.4 Этап сохранения и пост-обработки

После генерации ответа система выполняет ряд операций по сохранению результатов и обновлению внутренних структур.

Сохранение в историю диалогов происходит через SessionManager. Текущий диалог обновляется новым сообщением пользователя и ответом системы. Это позволяет поддерживать контекст для последующих запросов в рамках той же сессии.

Обновление FractalGraphV2 включает несколько операций. Из ответа извлекаются концепты и сохраняются как узлы графа с типами и связями. Если обнаружены противоречия с ранее сохранёнными концептами, они регистрируются для дальнейшей обработки. История диалогов также сохраняется для обеспечения долговременной памяти.

Обновление кэша токенов позволяет использовать результат для будущих запросов. Гибридная система кэширования сохраняет результат в горячий кэш (память GPU) для немедленного доступа и в холодный кэш (диск) для долгосрочного хранения.

## 3. Компоненты Системы и их Взаимодействие

### 3.1 UnifiedGenerator — ядро генерации

UnifiedGenerator является центральным компонентом системы генерации. Он инкапсулирует логику работы с тремя типами моделей и обеспечивает их координированное использование.

**SimpleRouter** выполняет интеллектуальный выбор модели на основе запроса. Роутер анализирует ключевые слова и характеристики запроса для определения наиболее подходящей модели. Для кодирования используется CODER модель, для кратких логических ответов — LOGIC, для развёрнутых объяснений — CONTEXT.

**_format_prompt** обеспечивает правильное форматирование входных данных для моделей. Система использует chat template с маркерами для идентификации ролей (system, user, assistant). Это позволяет моделям корректно интерпретировать структуру промта.

**_build_context** извлекает релевантную информацию из FractalGraphV2. Система выполняет семантический поиск по графу для нахождения концептов, связанных с текущим запросом. Найденный контекст включается в промт для повышения релевантности ответа.

### 3.2 FractalGraphV2 — система памяти

FractalGraphV2 реализует фрактальную структуру памяти, где информация хранится в иерархических узлах с множественными связями.

Узлы графа представляют концепты, диалоги и метаданные. Каждый узел имеет тип, контент и набор свойств. Связи между узлами определяют отношения типа «related_to», «contradicts», «supports» и другие.

Семантический поиск использует эмбеддинги для нахождения релевантных концептов. sentence-transformers модель преобразует текст в векторное представление, которое затем используется для поиска ближайших соседей в пространстве эмбеддингов.

### 3.3 Contradiction Manager — управление противоречиями

Contradiction Manager обнаруживает и отслеживает противоречия в знаниях системы.

Обнаружение противоречий выполняется через анализ связей в FractalGraphV2. Когда добавляется новый концепт, система проверяет существующие узлы на наличие противоположных утверждений. При обнаружении потенциального противоречия оно регистрируется для разрешения.

Разрешение противоречий может выполняться автоматически через самодиалог системы или через взаимодействие с пользователем. Система генерирует диалог между компонентами для выяснения природы противоречия и формирования согласованного ответа.

### 3.4 Ethics Framework — этический контроль

Ethics Framework обеспечивает фильтрацию нежелательного контента и соблюдение этических норм.

Проверка входящих сообщений анализирует текст на наличие запрещённых тем, потенциально опасного контента и запросов с этическими рисками. Система присваивает каждому сообщению оценку этичности и уровень риска.

Проверка исходящих сообщений выполняется после генерации ответа для убеждения, что сгенерированный контент соответствует этическим нормам. Это предотвращает распространение нежелательной информации.

## 4. Событийная Система и Асинхронная Обработка

### 4.1 EventBus — шина событий

EventBus обеспечивает коммуникацию между компонентами через асинхронные события. Это позволяет системе оставаться отзывчивой при выполнении длительных операций.

Основные события включают pipeline.start и pipeline.complete, сигнализирующие о начале и завершении генерации. System.state_changed уведомляет об изменении состояния системы. Component.initialized сообщает о готовности компонентов. Contradiction.detected сигнализирует об обнаружении противоречий.

### 4.2 Deferred Command System — отложенные команды

Deferred Command System управляет выполнением задач в фоновом режиме. Это позволяет разгрузить основной поток обработки запросов и обеспечить параллельное выполнение нескольких задач.

Система приоритетов определяет порядок выполнения команд. Критические операции (health checks, recovery) получают высокий приоритет. Фоновые задачи (оптимизация памяти, очистка) выполняются при низкой нагрузке.

Load shedding защищает систему от перегрузки. При высокой нагрузке на CPU или память низкоприоритетные команды отклоняются или откладываются. Это обеспечивает стабильность системы при пиковых нагрузках.

## 5. Поток Данных в Режиме Самодиалога

EVA-Ai способна к самообучению через механизм самодиалога. В этом режиме система инициирует диалог с собой для разрешения противоречий, извлечения новых концептов и обновления знаний.

### 5.1 Триггеры самодиалога

Самодиалог активируется при обнаружении противоречий в графе знаний. Когда Contradiction Manager обнаруживает конфликтующие утверждения, он инициирует диалог для их разрешения.

Концептуальное исследование также запускает самодиалог. При обнаружении новых концептов в запросе пользователя система инициирует исследовательский диалог для извлечения дополнительной информации и связей.

Периодическое самообучение запускается в фоновом режиме для обновления знаний. Система анализирует историю диалогов, извлекает новые концепты и обновляет структуру графа.

### 5.2 Структура самодиалога

Самодиалог реализует многосторонний диалог между различными аспектами системы. Assistant представляет текущую позицию, Critic анализирует противоречия, Learner исследует новые направления, Teacher формулирует рекомендации.

Результаты самодиалога сохраняются в FractalGraphV2 как новые узлы и связи. Это обогащает базу знаний системы и улучшает качество будущих ответов.

---

## 6. Заключение

Системный поток EVA-Ai представляет собой сложную, но элегантную архитектуру обработки информации. Ключевые принципы включают итеративное уточнение результатов, проверку согласованности, использование долговременной памяти и этический контроль на всех этапах обработки.

Трёхэтапная итеративная генерация (LOGIC → CONTEXT → LOGIC) обеспечивает баланс между краткостью и полнотой ответа, одновременно выполняя проверку на противоречия и извлечение концептов. Это отличает EVA-Ai от простых систем генерации текста и приближает её к концепции когнитивной системы.
