# CogniFlex (EVA-Ai)

## Когнитивная нейросетевая система нового поколения

**Версия:** 3.1  
**Дата:** 2026-04-06

---

## 1. Введение и философия системы

### 1.1 Обзор

**CogniFlex (EVA)** — это когнитивная система искусственного интеллекта, построенная на принципах рекурсивного рассуждения, фрактальной памяти и самообучения. В отличие от традиционных языковых моделей, EVA не просто генерирует текст — она анализирует, рассуждает, выявляет противоречия и постоянно совершенствуется.

### 1.2 Философия системы

EVA построена на фундаментальной идее: **мышление — это не генерация, а процесс**. Каждый ответ проходит через несколько стадий осмысления:

```
Запрос → Понимание → Анализ → Генерация → Проверка → Синтез → Ответ
```

Система не стремится выдать первый пришедший в голову ответ. Вместо этого она:

1. **Извлекает факты** — находит релевантные знания в памяти
2. **Анализирует** — проверяет логическую связность
3. **Развивает мысль** — добавляет контекст и детали
4. **Проверяет** — оценивает этичность и достоверность
5. **Синтезирует** — формирует финальный ответ

### 1.3 Отличие от традиционных LLM

| Характеристика | EVA (CogniFlex) | ChatGPT-4 | Claude | Gemini |
|---------------|-----------------|-----------|--------|--------|
| **Архитектура** | Three-GGUF Pipeline | Single model | Single model | Single model |
| **Память** | Фрактальный граф знаний | Контекстное окно | Контекстное окно | Контекстное окно |
| **Обучение** | Самодиалоги, автономное | RLHF (человеческая обратная связь) | RLHF | RLHF |
| **Проверка ответов** | 5+ этапов (этика, противоречия, качество) | Один уровень | Один уровень | Один уровень |
| **Fallback** | 7+ уровней при сбое | Один уровень | Один уровень | Один уровень |
| **Модульность** | Встроенные специализированные модули | Внешние API (плагины) | Ограниченно | Ограниченно |
| **Прозрачность** | Полная видимость процесса рассуждения | Частичная | Частичная | Частичная |
| **Автономность** | Самообучение без внешнего вмешательства | Требует обучения от разработчиков | Требует обучения от разработчиков | Требует обучения от разработчиков |

## 2. Ключевые инновации

### 2.1 Трёхуровневый конвейер генерации (Three-GGUF Pipeline)

Система использует три специализированные языковые модели Qwen2.5, каждая из которых выполняет свою функцию в процессе генерации:

```
┌─────────────────────────────────────────────────────────────┐
│                    ЗАПРОС ПОЛЬЗОВАТЕЛЯ                       │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Этап 1: Model A — Логическое ядро (Qwen2.5-3B)             │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ • Извлечение ключевых фактов из запроса             │     │
│  │ • Формирование краткого логичного ответа            │     │
│  │ • Консервативные параметры (temp=0.3)               │     │
│  │ • Макс. 1024 токенов, высокая точность              │     │
│  └─────────────────────────────────────────────────────┘     │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
                    ┌─────────┴─────────┐
                    │  Проверка         │
                    │  качества         │
                    └─────────┬─────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Этап 2: Model B — Концептуальное расширение (Qwen2.5-3B)   │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ • Развитие идей из Model A                         │     │
│  │ • Добавление примеров, деталей, контекста           │     │
│  │ • Строгий контроль повторений (repeat_penalty=2.0)  │     │
│  │ • Макс. 512 токенов                                 │     │
│  └─────────────────────────────────────────────────────┘     │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
                    ┌─────────┴─────────┐
                    │  Проверка         │
                    │  противоречий     │
                    └─────────┬─────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Этап 3: Model C — Кодогенератор (Qwen2.5-Coder-1.5B)       │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ • Ленивая загрузка (только при запросе кода)        │     │
│  │ • Специализированный промпт для кода                │     │
│  │ • Низкая температура (0.1) для точности             │     │
│  │ • Макс. 512 токенов                                 │     │
│  └─────────────────────────────────────────────────────┘     │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│              Финальная проверка и синтез                     │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ • Этическая проверка                                │     │
│  │ • Проверка противоречий с контекстом                │     │
│  │ • Сохранение в граф памяти                          │     │
│  └─────────────────────────────────────────────────────┘     │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      ОТВЕТ ПОЛЬЗОВАТЕЛЮ                      │
│  • Основной текст ответа                                     │
│  • Раскрываемое меню «Рассуждения» с этапами обработки       │
└─────────────────────────────────────────────────────────────┘
```

**Адаптивная настройка параметров:**

Система автоматически корректирует параметры генерации при обнаружении проблем:

| Проблема | Действие системы |
|----------|-----------------|
| Зацикливание (повторение фраз) | Увеличение repeat_penalty, снижение temperature |
| Пустой вывод | Увеличение max_tokens, изменение top_k |
| Нерелевантный ответ | Изменение промпта, повторная генерация |
| Высокая нагрузка CPU | Снижение max_tokens, уменьшение контекста |

**Семантическая детекция зацикливания:**

Система использует эмбеддинг-модель для сравнения новых генераций с предыдущими попытками. Если косинусное сходство превышает порог — генерация считается зацикленной и параметры корректируются.

### 2.2 Фрактальная память

EVA хранит знания в виде иерархического графа, где каждое понятие связано с другими через семантические ассоциации. Это не просто база данных — это структура, имитирующая ассоциативную память человека.

```
                    ┌──────────────┐
                    │    СНЕГ      │
                    │  (понятие)   │
                    └──────┬───────┘
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
        ▼                  ▼                  ▼
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│   белый      │   │  холодный    │   │   зимний     │
│  (цвет)      │   │ (температура)│   │   (сезон)    │
└──────┬───────┘   └──────┬───────┘   └──────┬───────┘
       │                  │                  │
       ▼                  ▼                  ▼
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ искрящийся   │   │   мороз      │   │   декабрь    │
│ (качество)   │   │  (явление)   │   │   (месяц)    │
└──────────────┘   └──────────────┘   └──────────────┘
```

**Типы семантических связей:**

| Тип связи | Пример | Описание |
|-----------|--------|----------|
| Атрибутивные | снег → белый, холодный | Свойства и характеристики |
| Контекстные | снег → зима, февраль | Связи по ситуации |
| Объектные | снег → сугроб, лыжи | Связанные объекты |
| Эмоциональные | снег → красивый, радостный | Эмоциональные ассоциации |
| Сенсорные | снег → искрящийся, пушистый | Ощущения |
| Каузальные | дождь → лужа, мокро | Причинно-следственные |

**Возможности фрактальной памяти:**
- Автоматическое создание связей между понятиями через куратор графа
- Извлечение контекста из предыдущих диалогов
- Визуализация графа знаний в веб-интерфейсе
- Гибридное кэширование для ускорения доступа

### 2.3 Система самодиалогов

EVA способна самостоятельно изучать новые понятия через процесс самодиалога:

```

│     │ • Разбиение на слова и словосочетания   │     │
│     └─────────────────────────────────────────┘     │
│                      │                              │
│                      ▼                              │
│  2. Проверка знаний                                 │
│     ┌─────────────────────────────────────────┐     │
│     │ • Поиск понятий в графе памяти          │     │
│     │ • Определение неизвестных концептов     │     │
```

**Пример работы:**
- Пользователь спрашивает о редком явлении
- EVA извлекает понятия из запроса
- Обнаруживает, что не знает один из терминов
- Выполняет веб-поиск по неизвестному понятию
- Сохраняет найденную информацию в граф памяти
- В следующий раз ответит на аналогичный вопрос из памяти

### 2.4 Интеллектуальные модули

Система включает специализированные модули, работающие параллельно с генерацией:

#### Модуль противоречий
- Сравнивает ответ с базой знаний
- Обнаруживает логические несоответствия
- Оценивает степень противоречия
- Запускает перегенерацию при обнаружении значимых противоречий

#### Модуль этики
- Проверяет запрос и ответ на соответствие этическим нормам
- Оценивает уровень риска
- Блокирует опасный контент до генерации
- Логирует все проверки для аудита

#### Модуль веб-поиска
- Автоматически активируется при низкой уверенности
- Пропускает простые запросы (приветствия, благодарности)
- Кэширует результаты для повторного использования
- Интегрирует найденную информацию в контекст

#### Модуль рассуждения
- Рекурсивный анализ сложных вопросов
- Многоуровневая проверка гипотез
- Self-reasoning engine для глубокого анализа

### 3.2 CoreBrain — центральный координатор

CoreBrain — тонкий координатор, управляющий системой через 8 миксинов:

| Миксин | Функция |
|--------|---------|
| ConfigMixin | Загрузка конфигурации, маскирование секретов |
| ComponentMixin | Жизненный цикл компонентов, управление модулями |
| QueryMixin | Многоуровневый fallback chain (7+ уровней) |
| MonitoringMixin | Health checks, системные метрики |
| MemoryMixin | Операции с фрактальной памятью |
| StateMixin | State machine (INITIALIZING → READY → RUNNING → SHUTTING_DOWN) |
| EventSubscriptionMixin | 13 обработчиков событий EventBus |
| CommandIssuerMixin | 20+ типов команд для управления системой |

**Жизненный цикл:**
1. `__init__()` — создание EventBus, DeferredCommandSystem
2. `initialize()` — инициализация всех компонентов, переход в READY
3. `start()` — запуск фоновых служб, переход в RUNNING
4. `stop()` — остановка, переход в SHUTTING_DOWN → OFFLINE
5. `reboot()` — полный цикл stop → reinitialize → start

### 3.3 EventBus — событийная шина

Централизованная система коммуникации между компонентами:

- **50+ типов событий** — системные, компоненты, обучение, генерация, память
- **Асинхронная обработка** — события ставятся в очередь, обрабатываются фоновым потоком
- **Автоматическая очистка** — удаление неактивных подписчиков через weak references
- **История событий** — кольцевой буфер на 10,000 событий

**Категории событий:**
```
system.*       — запуск, остановка, ошибки, готовность
component.*    — инициализация, ошибки компонентов
learning.*     — прогресс, завершение, сбои обучения
pipeline.*     — этапы генерации (model_a, model_b, complete)
generation.*   — прогресс, завершение, таймауты
memory.*       — предупреждения, оптимизация
contradiction.*— обнаружение и разрешение противоречий
ethics.*       — нарушения, предупреждения, проверки
```

### 3.4 DeferredCommandSystem — отложенные команды

Управление фоновыми задачами с приоритетами и защитой от перегрузки:

- **4 очереди приоритетов**: CRITICAL, HIGH, NORMAL, LOW
- **Повторные попытки** с экспоненциальной задержкой
- **Мониторинг здоровья** модулей (интервал 30 секунд)
- **Load shedding** — сброс низкоприоритетных задач при CPU > 80%

| Компонент | Назначение |
|-----------|------------|
| UnifiedFractalMemory | Единая фрактальная память с иерархической структурой |
| GraphLearning | Обучение на графе знаний |
| HybridTokenCache | Гибридный кэш токенов (RAM + Disk) |

### 3.6 Подсистема генерации

| Компонент | Назначение |
|-----------|------------|
| RecursiveModelPipeline | Three-GGUF Pipeline (Model A → B → C) |
| AdaptiveParameterController | Семантическая детекция зацикливания, адаптация |
| QualityChecker | Детекция повторений, filler фраз, китайских символов |
| GenerationTracker | Трекинг жизненного цикла генераций с SSE |
| **UnifiedCacheBridge** | **Объединённый кэш модели и графа знаний** |

**UnifiedCacheBridge** (`core/unified_cache_bridge.py`):
- Семантический поиск релевантных узлов графа знаний по запросу
- Предзагрузка узлов в HybridTokenCache (VRAM → RAM → Disk)
- Обогащение промптов контекстом из графа
- Кэширование результатов генерации — мгновенные ответы на повторные запросы
- Автоматическая инициализация в ResponseGenerator после HybridTokenCache

### 3.7 Веб-интерфейс

- Тёмная тема с CSS custom properties
- Управление сессиями (создание, переименование, удаление)

### 3.8 Безопасность

- **Аутентификация**: PBKDF2-SHA256, 100,000 итераций, случайная соль
- **Сессии**: Thread-safe хранение с лимитами (20 context nodes, 50 chat history)
- **Загрузка файлов**: UUID-именование, поддержка 20+ форматов
- **API защита**: Проверка токенов, rate limiting

### 4.2 Обработка ошибок и fallback

Система имеет многоуровневую систему fallback:

```
Запрос → Primary Pipeline (Three-GGUF)
              │
              ▼ (сбой)
         Fallback 1: Single GGUF генерация
              │
              ▼ (сбой)
         Fallback 2: SelfReasoningEngine
              │
              ▼ (сбой)
         Fallback 3: EnhancedReasoningEngine
              │
              ▼ (сбой)
         Fallback 4: MLUnit
              │
              ▼ (сбой)
         Fallback 5: Memory-based response
              │
              ▼ (сбой)
         Fallback 6: Keyword-based response
              │
              ▼ (сбой)
         Fallback 7: Базовое сообщение "Не могу ответить"
```

### 4.3 Кэширование на этапах

| Этап | Тип кэша | Время жизни |
|------|----------|-------------|
| Эмбеддинги | EmbeddingCache (SQLite LRU) | До заполнения |
| Токены генерации | HybridTokenCache (RAM + Disk) | Настраиваемый |
| Результаты веб-поиска | Search Cache (JSON) | 24 часа |
| Контекст сессии | Session Cache (JSON) | До удаления сессии |

## 5. Безопасность и этика

### 5.1 Этические принципы EVA

Система руководствуется шестью ключевыми принципами:

| # | Принцип | Описание |
|---|---------|----------|
| 1 | **Без насилия** | Не отвечает на вопросы о насилии, оружии, наркотиках, травле |
| 2 | **Честность** | Если не уверена — говорит об этом, просит уточнение |
| 3 | **Проверка фактов** | Проверяет информацию, исправляет ошибки, не выдаёт догадки за факты |
| 4 | **Безопасный код** | Не генерирует вредоносный или деструктивный код |
| 5 | **Блокировка рисков** | Отклоняет высокорисковые запросы, предупреждает о средних рисках |
| 6 | **Контроль вывода** | Проверяет ответ на этичность перед отправкой |

### 5.2 Модуль безопасности

- **Проверка входящих запросов** — анализ на опасные паттерны
- **Анализ intent** — определение намерений пользователя
- **Оценка уровня риска** — классификация по уровням (низкий, средний, высокий)
- **Логирование** — запись всех заблокированных запросов для аудита

### 5.3 Детекция опасного контента

Система проверяет запросы и ответы на:
- Насилие и угрозы
- Запросы на создание вредоносного ПО
- Запрещённые темы (наркотики, оружие)
- Кибербуллинг и преследование
- Дезинформацию и пропаганду

### 6.1 Аппаратные требования

| Компонент | Минимум | Рекомендуется | Назначение |
|-----------|---------|---------------|------------|

| SSD | 10 ГБ свободно | 50+ ГБ SSD | GGUF модели, кэши, граф памяти |

| Метрика | Значение |
|---------|----------|

| Веб-поиск | 1-2 секунды |
| Контекстное окно | до 4096 токенов |
| Куратор графа | Фоновый режим, каждые 5 минут |
| Кэш-попадание токенов | Зависит от повторяемости запросов |

### 6.3 Оптимизации

- **Q4_K_M квантование** — 4-битное сжатие моделей для экономии памяти
- **Ленивая загрузка** — Model C загружается только при запросе кода
- **Адаптивное снижение ресурсов** — уменьшение max_tokens при высокой нагрузке
- **Гибридное кэширование** — RAM для горячих данных, Disk для холодных

## 9. Разработка и расширение

### 9.1 Модульная архитектура

Система построена на принципах модульности:

```
eva/
├── core/           # Ядро (EventBus, Pipeline, Commands, CoreBrain)
├── memory/         # Фрактальная память и кэши
├── mlearning/      # ML модели и управление
├── knowledge/      # Граф знаний, Wikipedia
├── learning/       # Самообучение, самодиалоги
├── websearch/      # Веб-поиск (Tavily API)
├── ethics/         # Этический фреймворк
├── contradiction/  # Детекция и разрешение противоречий
├── reasoning/      # Рекурсивное рассуждение
├── generation/     # Координация генерации
├── gui/            # Веб-интерфейс
├── security/       # Фреймворк безопасности
├── monitoring/     # Мониторинг системы
├── recovery/       # Система восстановления
└── adaptation/     # Адаптивные системы
```

### 9.2 Принципы разработки

| Принцип | Описание |
|---------|----------|
| **Модульность** | Каждый компонент — отдельный модуль с чётким интерфейсом |
| **Изоляция** | Ошибки в одном модуле не ломают систему (fallback chain) |
| **Расширяемость** | Новые модули добавляются без изменения ядра |
| **Тестируемость** | Каждый модуль можно тестировать отдельно |
| **Событийность** | Коммуникация через EventBus, а не прямые вызовы |

### 9.3 Добавление нового модуля

Для добавления нового модуля необходимо:

1. Создать директорию в `eva/` с именем модуля
2. Реализовать основной класс с методами инициализации
3. Зарегистрировать модуль в `brain_components.py`
4. Подписаться на нужные события через EventBus
5. Добавить обработку команд через DeferredCommandSystem
