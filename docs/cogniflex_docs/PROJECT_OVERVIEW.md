# CogniFlex: Обзор проекта

## Краткое описание
CogniFlex — модульная платформа для интеллектуальной обработки текста и управления знаниями. Система фокусируется на:
- обнаружении и разрешении противоречий в знаниях
- контроле качества ответов и устойчивости NLP-пайплайна к отсутствующим опциональным зависимостям
- сборе метрик производительности и здоровья системы
- упрощённой интеграции с обучающими и сервисными компонентами

## Структура проекта
- `cogniflex/`
  - `contradiction/`
    - Логика анализа и разрешения противоречий.
    - Ключевые файлы: `contradiction_strategies.py`, `contradiction_learning.py`.
  - `core/`
    - Базовая логика ядра (инициализация, маршрутизация, системные сервисы). Например, `core_brain.py` с `CoreBrain._init_nlp_inference()`.
  - `adapters/`
    - Слоёвые адаптеры для опциональных зависимостей (например, Torch).
  - `adaptation/`
    - Аналитика адаптации и вспомогательные механизмы.
- `tests/`
  - Юнит- и интеграционные тесты (54 теста; покрытие ключевых сценариев, smoke и производительность).
- `tools/`
  - Скрипты проверки/бенчмаркинга, финальные smoke-тесты для окружения.
- Кэш/модели/служебные каталоги: `cogniflex_cache/`, `ml_cache/`, `hybrid_cache/`, `nltk_data/`, и т.п.
- Документация: `docs/` (API, архитектура, деплой).

## Ключевые достижения реализации
- Защита опциональных NLP-зависимостей в модулях противоречий:
  - В `cogniflex/contradiction/contradiction_strategies.py` вызовы `nlp_model.encode` обёрнуты в `try/except` с фоллбеком на TF‑IDF + `cosine_similarity`.
  - Инициализация `SentimentIntensityAnalyzer` выполнена с кешированием на уровне стратегии и с безопасными фоллбеками.
  - В `_calculate_concept_importance()` защищены стоп-слова/токенизация NLTK; предусмотрены простые замены без данных NLTK.
- Унификация и отладка эмиссии метрик (event-first с единым фоллбеком) для консистентности и отсутствия дубликатов.
- Тестовый прогон: 54 теста пройдены, предупреждения носят некритичный характер (Click 9.0 депрекации, нестандартные pytest-маркеры, стилистика возврата значений в тестах).
- Сохранение устойчивости системы при отсутствии `sentence_transformers`/`torch` и NLTK-данных.

## Принципы функционирования
- Устойчивость по умолчанию:
  - Любой опциональный компонент (модель/библиотека) не должен ломать основной поток.
  - Всегда есть фоллбек (TF-IDF, простая токенизация/стоп-слова, «мягкие» метрики).
- Наблюдаемость:
  - Единый шаблон эмиссии метрик с приоритетом события и безопасным фоллбеком.
  - Логирование ошибок без прерывания основного хода.
- Модульность и расширяемость:
  - Чёткие границы между ядром (`core/`), доменом противоречий (`contradiction/`), адаптерами (`adapters/`).
  - Возможность добавлять стратегии и правила без изменения базы.
- Производительность разумного уровня:
  - Локальное кеширование анализаторов и минимизация повторных инициализаций.
  - Лёгкие фоллбеки для долгих/тяжёлых операций при отсутствии зависимостей.

## Технологии и зависимости
- Python 3.13 (виртуальное окружение `.venv/`).
- Опционально: `sentence_transformers`, `torch` (используются, если доступны).
- Фоллбек‑методы: `scikit-learn` (TF‑IDF, cosine similarity), NLTK (если доступна), с безопасными заменами.
- Тестирование: `pytest` (+ вспомогательные скрипты в `tools/`).

## Потоки данных (упрощённо)
1. `CoreBrain` инициирует NLP‑инференс и сервисы.
2. Входные факты/запросы → модуль `contradiction/`.
3. Детекция/оценка противоречий → расчёт метрик (семантика/лексика/тональность) с фоллбеками.
4. Рекомендация стратегии разрешения → генерация плана обучения при необходимости (`contradiction_learning.py`).
5. Эмиссия метрик и обновление графа знаний/кэшей.

## Качество и тестирование
- 54 теста «зелёные» на текущей ревизии.
- Предупреждения:
  - `PytestUnknownMarkWarning` (не зарегистрирован `timeout`) — можно добавить `pytest.ini`.
  - `PytestReturnNotNoneWarning` — стилистика тестов, не критично.
  - Депрекации Click 9.0 из библиотек — информационно.

## Дальнейшие улучшения (рекомендации)
- Добавить `pytest.ini` с регистрацией пользовательских маркеров (например, `timeout`).
- Небольшие оптимизации производительности: переиспользование TF‑IDF‑векторизатора в горячих путях при пакетной обработке.
- Документировать контракт интерфейса `nlp_model` (ожидаемое наличие `encode`).
- Сформировать мини‑гайд по политике опциональных зависимостей и фоллбеков.
