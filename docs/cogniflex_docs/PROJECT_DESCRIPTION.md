# CogniFlex — Подробное описание проекта (актуализировано)

CogniFlex — модульная когнитивная платформа с акцентом на устойчивость, наблюдаемость и эффективное управление ресурсами. Система объединяет ядро управления, ML‑подсистему, гибридное кэширование токенов (RAM+диск) и сервисы метрик, а также готовится к миграции GUI на PySide6.

## Ключевые компоненты

- __Core (`cogniflex/core/`)__
  - `core_brain.py` — центральный координатор: инициализация, запуск/останов, статусы модулей, метрики, обработка запросов, безопасные фолбэки. Экспортирует удобные обёртки для системного арбитра ресурсов: `request_memory()`, `release_memory()`, `request_cpu()`, `release_cpu()`.
  - `global_resource_queue.py` — GlobalResourceQueue (GRQ): системный арбитр RAM/CPU/IO, обеспечивает справедливое распределение и троттлинг IO.
  - Служебные утилиты, фоны и политики — см. `background_coordinator`, `device_resolver`, менеджеры состояния/метрик при наличии.

- __Memory (`cogniflex/memory/`)__
  - `hybrid_token_cache.py` — гибридный кэш токенов: быстрый LRU в памяти + устойчивый диск‑кэш с троттлингом IO. Интегрирован с GRQ для строгого учёта RAM‑окон при размещении в памяти и корректного освобождения при вытеснениях.
  - `disk_cache.py` — устойчивый диск‑кэш с SQLite‑метаданными и локальным токен‑бакетом на чтение/запись, дополнительно учитывает системный троттлинг GRQ, если он подключён.

- __ML (`cogniflex/mlearning/`)__
  - `model_manager.py`, `unified_text_processor.py` — управление моделями и унифицированная обработка текста с офлайн‑фолбэками, событийной интеграцией и асинхронной токенизацией (если доступно).

- __Документация (`docs/`)__
  - `architecture.md`, `deployment.md`, `api_reference.md` — справочные материалы. Текущий файл дополняет их детальной, актуализированной картиной.

## Управление ресурсами: GlobalResourceQueue (GRQ)

- __RAM__: оконная модель с очередью ожидания и простым aging, исключающая «тихие» утечки. Методы: `acquire_memory(nbytes, timeout=None)`, `release_memory(nbytes)`.
- __CPU__: счётчики токенов CPU с очередью ожидания: `acquire_cpu(n, timeout=None)`, `release_cpu(n)`.
- __IO__: блокирующий троттлинг по скорости (token‑bucket с burst‑ёмкостью): `acquire_io(nbytes)`.
- __Диагностика__: `snapshot()` возвращает консистентный срез состояния RAM/CPU/IO для метрик/GUI.

## Гибридный кэш токенов

- __Дизайн__: 
  - Данные всегда сохраняются на диск для достижения высокой утилизации дискового кэша (~50GB по умолчанию).
  - Дублирование «горячих» токенов в RAM выполняется только после успешного запроса RAM‑окна через GRQ. При переполнении — явное LRU‑вытеснение с освобождением ранее занятого RAM‑окна.
- __Безопасность и стабильность__: 
  - Явное `_evict_one_lru()` перед вставкой в память гарантирует вызов `release_memory()` и отсутствие «тихих» утечек RAM‑окон.
  - Троттлинг IO при чтении/записи в `DiskCache` локально и через GRQ.
- __Наблюдаемость__: сбор статистики попаданий в память/на диск, среднее время доступа, эффективность кэша и т.д.

## Потоки и сценарии

1. `CoreBrain` инициализирует компоненты и (при наличии) `resource_queue` (GRQ).
2. `HybridTokenCache` при `add_token()` всегда пишет на диск и, при наличии окна RAM, добавляет в память, учитывая LRU‑вытеснение и корректный учёт RAM.
3. При чтении `get_token()` данные берутся из RAM или диска. «Горячие» токены могут фоново мигрировать в память c запросом RAM‑окна.
4. `DiskCache` сериализует+сжимает записи, ведёт SQLite‑метаданные, применяет локальный и системный троттлинг IO.

## Надёжность и офлайн‑режим

- В ML‑части и противоречиях предусмотрены фолбэки при отсутствии тяжёлых зависимостей или данных.
- В ядре — защитные шаблоны логирования/метрик, чтобы исключить падение конвейера при частичных сбоях.
