# Анализ результатов тестирования FractalQwenGenerator (FMF)

Полученные данные демонстрируют работоспособность всех ключевых компонентов системы, но выявили значительное замедление при включении графового контекста (L2/L3). Ниже представлен детальный разбор, причины замедления и конкретные рекомендации по оптимизации.

---

## 1. Детализация результатов

| Компонент | Статус | Наблюдение |
|-----------|--------|------------|
| **L0 Pipeline** | ✅ 27 сек | Базовая генерация модели Qwen3 4B на CPU (ожидаемо для 4B). |
| **L1 Profiler** | ✅ 2 активации | Успешно созданы/обновлены профили активаций. |
| **L2 Router** | ✅ 2 правила | Правила маршрутизации загружены и применены. |
| **L3 Graph** | ✅ 7 узлов | Граф содержит 7 узлов (включая `concept_ai`). |

**Ключевой показатель:**
- **Без графа:** 27618 мс (≈27.6 сек)
- **С графом (L2):** 69575 мс (≈69.6 сек)
- **Замедление:** в **2.52 раза**

---

## 2. Причины значительного замедления при включении графа

### 2.1. Гибридная токенизация (Aho‑Corasick)

- При каждом запросе выполняется проход Aho‑Corasick автомата по тексту запроса. Время линейно зависит от длины запроса и числа узлов в автомате. Для 7 узлов — пренебрежимо мало (<< 1 мс).
- **Вывод:** не является источником задержки.

### 2.2. Вычисление эмбеддингов запроса (SentenceTransformer)

- Для определения домена и поиска правил L2 вычисляется эмбеддинг запроса.
- Модель `intfloat/multilingual-e5-large` (1024‑мерный эмбеддинг) на CPU может занимать **50–150 мс** на один текст.
- **Влияние:** минимальное (~0.2% от 70 сек).

### 2.3. Семантический поиск в графе

- Поиск ближайших узлов по эмбеддингу (линейный проход по 7 узлам) — пренебрежимо.

### 2.4. Проекция эмбеддингов (1024 → 3584)

- Операция `np.tile` с нормировкой — доли миллисекунд.

### 2.5. **Основная причина: двойной прогон инференса при использовании L2**

- В текущей реализации `RoutingEngine` может **дважды вызывать модель**:
  1. Для получения эмбеддинга запроса (если используется модель в режиме энкодера).
  2. Для генерации ответа.
- Однако в коде мы используем отдельный `SentenceTransformer`, поэтому двойного прогона LLM быть не должно.

### 2.6. **Наиболее вероятная причина: синхронная запись в SQLite во время инференса**

- После генерации запускается **асинхронный** поток для обновления L1.
- **Но** если в коде присутствует вызов `graph.update_activation_profile()` **синхронно** до возврата ответа, это приводит к блокировке на дисковой операции (SQLite `UPDATE` с транзакцией).
- Время записи в SQLite на HDD может составлять десятки миллисекунд, но на SSD — единицы.
- Однако 40 секунд разницы указывают на нечто более серьёзное.

### 2.7. **Критическая гипотеза: повторная загрузка/применение LoRA‑адаптера**

- Если при каждом запросе `LoRAManager` загружает адаптер с диска и применяет его через `pipe.set_adapters()`, это может занимать секунды.
- Проверьте, не вызывается ли `load_adapter()` и `apply_adapter()` синхронно внутри `generate()` без кэширования.

### 2.8. **Возможная причина: использование enable_thinking в режиме с графом**

- Флаг `enable_thinking` в Qwen3 включает генерацию цепочки рассуждений, что увеличивает длину выхода и время инференса.
- Если для запроса с графом случайно включился `enable_thinking`, а для базового — нет, это объяснит замедление.

### 2.9. **Проблема с OpenVINO stateful/динамическими формами**

- При изменении параметров генерации (например, `max_new_tokens`) OpenVINO может перекомпилировать граф, что занимает секунды.

## 3. Пошаговая диагностика и оптимизация

### Шаг 1. Профилирование времени выполнения

Добавьте временные метки в ключевые точки `generate()`:

```python
import time
def generate(...):
    t0 = time.time()
    tokens, embs = self.hybrid_tokenizer.encode(...)
    t1 = time.time()
    domain = self._extract_domain_from_tokens(tokens)
    t2 = time.time()
    config = self.router.get_generation_config(...)
    t3 = time.time()
    # LoRA
    if use_lora:
        alias = self.lora_manager.load_adapter(domain)
        if alias:
            self.lora_manager.apply_adapter(alias, alpha=0.8)
    t4 = time.time()
    # build prompt
    enriched = self._build_context_prompt(...)
    t5 = time.time()
    response = self.pipe.generate(...)
    t6 = time.time()
    # async update...
    print(f"Tokenize: {t1-t0:.3f}s, Domain: {t2-t1:.3f}s, Router: {t3-t2:.3f}s, LoRA: {t4-t3:.3f}s, Prompt: {t5-t4:.3f}s, Inference: {t6-t5:.3f}s")
```

### Шаг 2. Проверка LoRA‑менеджера

Убедитесь, что адаптер загружается один раз и кэшируется:

```python
def load_adapter(self, domain: str) -> Optional[str]:
    if domain in self._cache:
        return domain  # уже в кэше, не грузим с диска
    adapter_path = self.adapters_dir / f"{domain}_lora"
    if not adapter_path.exists():
        return None
    # Загружаем и сохраняем в кэш
    self._cache[domain] = ov_genai.Adapter(str(adapter_path))
    return domain
```

### Шаг 3. Отложенное обновление L1/L2

Убедитесь, что обновление профиля действительно асинхронное:

```python
threading.Thread(target=self._update_profiles, args=(...), daemon=True).start()
```

И что внутри `_update_profiles` нет вызовов, блокирующих основной поток.

### Шаг 4. Проверка `enable_thinking`

Убедитесь, что флаг `enable_thinking` передаётся одинаково в обоих тестах. Для чистоты сравнения отключите его явно:

```python
enable_thinking=False
```

### Шаг 5. Кэширование `apply_chat_template`

`tokenizer.apply_chat_template` может быть медленным. Кэшируйте шаблон для повторяющихся структур.

### Шаг 6. Использование батч-инференса для L1 (если возможно)

Вычисление эмбеддинга запроса и ответа можно делать один раз и использовать для обновления профиля позже.

## 4. Рекомендации по архитектурным улучшениям

### 4.1. Отказ от LoRA в коротких запросах

Для запросов, где домен определён, но адаптер даёт минимальный прирост качества, можно отключать LoRA, чтобы сэкономить время на его применении (хотя оно должно быть малым).

### 4.2. Использование быстрого энкодера

Заменить `multilingual-e5-large` на более лёгкую модель (например, `paraphrase-multilingual-MiniLM-L12-v2` или даже `all-MiniLM-L6-v2`), которая даёт 384‑мерные эмбеддинги и работает в 3–5 раз быстрее. Проектор адаптировать под новую размерность.

### 4.3. Кэширование эмбеддингов запросов

Если пользователь задаёт уточняющие вопросы по той же теме, можно кэшировать эмбеддинг первого запроса и использовать его для домена.

### 4.4. Предварительная компиляция модели с динамическими формами

В OpenVINO можно указать диапазон возможных `max_new_tokens` при загрузке, чтобы избежать рекомпиляции при изменении параметра.

### 4.5. Вынос графа в оперативную память

Если граф находится на медленном диске, загрузите SQLite базу в `:memory:` при старте (или используйте `PRAGMA journal_mode=MEMORY`).

---

## 5. Ожидаемый результат после оптимизации

После устранения выявленных узких мест время генерации с графом должно приблизиться к времени базовой генерации плюс небольшая константная добавка (50–200 мс) на обработку графа и токенизацию. **Целевое замедление не более 5–10%.**
