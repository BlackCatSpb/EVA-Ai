## 2. Выявленные проблемы

### 2.1. Ошибка SQLite: "Recursive use of cursors not allowed"

```
File "fmf_optimized.py", line 189, in get_concepts
    self.cursor.execute("SELECT id, content FROM nodes WHERE node_type='concept'")
sqlite3.ProgrammingError: Recursive use of cursors not allowed.
```

**Причина:** В многопоточной среде (асинхронное обновление L1) происходит попытка выполнить SQL-запрос через тот же объект `cursor`, который уже используется в другом потоке или рекурсивно внутри того же соединения. SQLite не разрешает рекурсивное использование курсоров в одном соединении.

**Решение:**
- Создавать отдельное соединение с базой данных для каждого потока.
- Либо использовать пул соединений.
- Либо защитить доступ к курсору блокировкой (`threading.Lock`).

**Пример исправления:**

```python
import sqlite3
import threading

class FractalGraphV2:
    def __init__(self, db_path: str):
        self.db_path = db_path
        self._local = threading.local()
    
    def _get_connection(self):
        if not hasattr(self._local, 'conn'):
            self._local.conn = sqlite3.connect(self.db_path)
            self._local.conn.row_factory = sqlite3.Row
        return self._local.conn
    
    def get_concepts(self):
        conn = self._get_connection()
        cursor = conn.cursor()
        cursor.execute("SELECT id, content FROM nodes WHERE node_type='concept'")
        return cursor.fetchall()
```

### 2.2. Виртуальные токены не найдены (Virtual tokens: 0)

В обоих тестах гибридный токенизатор не обнаружил совпадений с узлами графа, несмотря на то что запросы содержат концепты "искусственный интеллект" (Test 2). Возможные причины:

1. **Регистр или точное совпадение**  
   В `HybridTokenizer._build_automaton()` текст узла приводится к нижнему регистру, и поиск также ведётся по `text.lower()`. Это должно работать.

2. **Неполное совпадение**  
   Если в графе узел называется `"искусственный интеллект"`, а в запросе `"Расскажи про искусственный интеллект"`, автомат должен найти подстроку.

3. **Проблема с построением автомата**  
   Возможно, узлы не были добавлены в автомат из-за фильтрации по типу или слою.

**Диагностика:**

Добавить отладочный вывод в `HybridTokenizer.encode()`:

```python
def encode(self, text: str, layers: Set[int] = {2, 3}, return_embeddings: bool = False):
    print(f"[Tokenizer] Searching in: '{text.lower()}'")
    matches = list(self.automaton.iter(text.lower()))
    print(f"[Tokenizer] Found {len(matches)} matches")
    for end, (tid, matched) in matches:
        print(f"  - '{matched}' -> token {tid}")
    ...
```

**Вероятная причина:** В текущей реализации `encode()` используется `self.automaton.iter(text_lower)`, но из-за особенностей `ahocorasick` метод `iter()` возвращает кортежи `(end_index, value)`, где `value` — это то, что было добавлено (`(token_id, text)`). В коде ранее мы извлекали `token_id, matched_text = value`. Это корректно.

Необходимо убедиться, что автомат действительно содержит узлы. Вывести размер автомата после построения:

```python
print(f"[Tokenizer] Automaton size: {len(self.automaton)}")
```

### 2.3. Время генерации остаётся высоким

24–39 секунд — это ожидаемая цена за качество при включённом `enable_thinking`. Модель генерирует не только ответ, но и подробную цепочку рассуждений (блок `<think>`), что увеличивает количество токенов в 2–3 раза.

Если требуется ускорение без потери качества, можно рассмотреть:

- **Квантизацию модели** до INT8 или INT4 (уже используется `openvino_genai` с оптимизированными весами, но можно проверить, какая точно квантизация применена).
- **Использование GPU** (OpenVINO поддерживает GPU-ускорение).
- **Сокращение длины размышлений** путём настройки промпта (например, указать "будь краток в рассуждениях").
