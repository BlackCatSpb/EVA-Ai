# FractalQwenGenerator: Детальное описание объединённой архитектуры

## 1. Концептуальная модель

**FractalQwenGenerator** — это генеративная система, объединяющая языковую модель и фрактальный граф знаний в единую когнитивную архитектуру. Ключевая идея: знания хранятся в многослойном графе (L0–L3), а модель получает к ним прямой доступ через гибридный токенизатор и динамические LoRA‑адаптеры. Система способна работать в трёх режимах:

- **Диалоговый режим** — короткие запросы с автоматической адаптацией параметров.
- **Контекстный режим** — временная загрузка документов как расширение графа.
- **Пакетный режим** — обработка больших текстов с сохранением структуры.

## 3. Слоистая структура (L0–L3)

### 3.1. L0 — Инференс

- **Компоненты:** OpenVINO GenAI Pipeline, LoRAManager.
- **Функции:**
  - Загрузка и выполнение модели Ruadapt Qwen3 4B.
  - Динамическое применение LoRA‑адаптеров для специализации на доменах.
  - Поддержка флага `enable_thinking` для включения цепочки рассуждений.
- **Вход:** промпт (строка), параметры генерации (`temperature`, `top_p`, `max_new_tokens`, `repetition_penalty`).
- **Выход:** сгенерированный текст.

### 3.2. L1 — Профили активаций

- **Компонент:** `ActivationProfiler`.
- **Функции:**
  - Сбор статистики по каждому запросу: латентность, энтропия, качество.
  - Вычисление **fingerprint** — усреднённого эмбеддинга пары (запрос, ответ).
  - Инкрементальное обновление центроида домена в графе (узел `activation_profile`).
  - Временной распад старых профилей (через `domain_lambda`).
- **Хранение:** узлы типа `activation_profile` в графе L3.

### 3.3. L2 — Маршрутизация

- **Компонент:** `RoutingEngine`.
- **Функции:**
  - Определение оптимальных параметров генерации для запроса.
  - Поиск правила `routing_rule` по домену (точное совпадение или через семантический поиск профилей).
  - Кэширование конфигураций для часто используемых доменов.
  - Обновление статистики успешности правил (при обратной связи).
- **Хранение:** узлы типа `routing_rule`, связанные с концептами доменов.

### 3.4. L3 — Когнитивная память

- **Компонент:** `FractalGraphV2`.
- **Постоянная часть:**
  - `concept` — ключевые понятия.
  - `fact` — утверждения, связанные с концептами.
  - `aci_concept` — концепты, порождённые автономным выводом (ACI).
  - `routing_rule` — правила маршрутизации (L2).
  - `activation_profile` — профили активаций (L1).
- **Временная часть (сессии):**
  - `temp_paragraph` — параграфы загруженного документа.
  - `temp_entity` — извлечённые сущности (опционально).
  - Связи между временными узлами (`next`, `parent`).
- **Операции:** добавление/удаление узлов, семантический поиск, навигация по рёбрам.

## 7. Преимущества объединённой архитектуры

| Аспект | Преимущество |
|--------|--------------|
| **Единое хранилище** | Модель, граф и метаданные в одном файле `.fmf` — упрощает развёртывание и обмен. |
| **Гибридная токенизация** | Прямой доступ модели к знаниям графа через виртуальные токены без раздувания промпта. |
| **Адаптивность** | Автоматический подбор параметров генерации под домен запроса (L2) и обновление профилей (L1). |
| **Контекстная работа с документами** | Временные подграфы позволяют задавать вопросы по документу без повторной загрузки. |
| **Масштабируемость** | Батчевая обработка с сохранением структуры через специальные токены. |
| **Самообучение** | LoRA‑адаптеры «вживляют» знания графа в веса модели, улучшая качество с течением времени. |
| **Отказоустойчивость** | Временные сессии автоматически очищаются, постоянный граф не затрагивается. |

## Сценарий 1. Инициализация системы

### Шаг 1. Загрузка модели и токенизатора
1. Система принимает путь к директории с OpenVINO IR‑моделью (`.xml` и `.bin` файлы).
2. Создаётся экземпляр `ov_genai.LLMPipeline` — основной движок инференса.
3. Параллельно загружается токенизатор `AutoTokenizer` из той же директории (специфичный для Qwen3, поддерживает `apply_chat_template` и флаг `enable_thinking`).
4. Устанавливается устройство выполнения (`CPU`, `GPU` или `NPU`).

### Шаг 2. Загрузка энкодера и проектора
1. Инициализируется модель `SentenceTransformer` (например, `intfloat/multilingual-e5-large`) для получения эмбеддингов размерности 1024.
2. Создаётся `EmbeddingProjector`, преобразующий 1024‑мерные векторы в 3584‑мерное пространство скрытого состояния Qwen3 (метод `repeat` или `linear`).

### Шаг 3. Загрузка фрактального графа
1. Открывается SQLite‑файл графа (`FractalGraphV2`).
2. Регистрируются специальные типы узлов: `activation_profile`, `routing_rule`.
3. Проверяется наличие индексов для быстрого семантического поиска.

### Шаг 4. Построение гибридного токенизатора
1. Из графа извлекаются все текстовые метки узлов слоёв L2 и L3 (концепты, факты, правила).
2. Строится **Aho‑Corasick автомат** — структура для одновременного поиска множества подстрок.
3. Для каждого найденного узла создаётся **виртуальный токен** с уникальным ID (начиная с 151936). Сохраняется его эмбеддинг, тип и принадлежность слою.
4. Гибридный токенизатор готов к замене подстрок запроса на виртуальные ID.

### Шаг 5. Инициализация L1 и L2 сервисов
1. `ActivationProfiler` получает ссылки на граф, энкодер и проектор.
2. `RoutingEngine` получает ссылки на граф и энкодер; кэш правил маршрутизации пуст.

### Шаг 6. Загрузка LoRA‑менеджера
1. Указывается директория с предобученными адаптерами (формат safetensors).
2. `LoRAManager` готов динамически загружать и применять адаптеры через `ov_genai.AdapterConfig`.

## Сценарий 2. Короткий диалоговый запрос

### Шаг 1. Предобработка запроса гибридным токенизатором
1. Пользователь вводит текст, например: *«Расскажи про фотосинтез»*.
2. Текст приводится к нижнему регистру и подаётся в Aho‑Corasick автомат.
3. Автомат находит совпадения с узлами графа:
   - `"фотосинтез"` → узел типа `concept` (L3), ID виртуального токена 152345.
4. Токенизатор формирует выходной список:
   - BPE‑токены для префикса `"Расскажи про "`
   - Виртуальный токен `152345`
   - BPE‑токены для оставшейся части (если есть).
5. Опционально возвращается массив эмбеддингов найденных узлов (спроецированных в 3584).

### Шаг 2. Определение домена запроса
1. Анализируются использованные виртуальные токены. Если среди них есть тип `concept`, берётся его текст (`"фотосинтез"`).
2. Дополнительно может быть вычислен эмбеддинг всего запроса через `SentenceTransformer` и спроецирован в 3584.
3. Итоговый домен: `"биология"` (или непосредственно `"фотосинтез"`, если такой концепт существует в графе).

### Шаг 3. Получение параметров генерации от RoutingEngine (L2)
1. `RoutingEngine.get_generation_config()` вызывается с аргументом `domain_hint="биология"`.
2. Проверяется кэш правил: если домен уже встречался, возвращается сохранённая конфигурация.
3. Иначе выполняется поиск в графе:
   - Ищется узел типа `concept` с содержимым `"биология"`.
   - По связи `applies_to_domain` находится `routing_rule`.
   - Из `metadata` правила извлекаются `temperature`, `top_p`, `repetition_penalty`, `max_new_tokens`.
4. Если правило не найдено, используются значения по умолчанию (`temperature=0.2`, `top_p=0.9` и т.д.).
5. Возвращается объект `GenerationConfig`.

### Шаг 4. Применение LoRA‑адаптера (если включено)
1. Если `use_lora=True` и домен не `"general"`, `LoRAManager` ищет адаптер в директории `./lora_adapters/{domain}_lora`.
2. При наличии загружает его (кэширует) и применяет через `pipe.set_adapters()`.
3. Адаптер модифицирует веса модели «на лету», специализируя её на предметной области.

### Шаг 5. Построение обогащённого промпта
1. Из графа извлекаются до трёх фактов (`fact`), связанных с концептом домена (например, *«фотосинтез происходит в хлоропластах»*).
2. Формируется текстовый блок: `"Используй знания:\n- фотосинтез происходит в хлоропластах\n\nВопрос: Расскажи про фотосинтез\nОтвет:"`.
3. Если переданы `context_nodes`, они добавляются принудительно.

### Шаг 6. Формирование полного промпта для Qwen3
1. Вызывается `tokenizer.apply_chat_template` с сообщением `[{"role": "user", "content": enriched_prompt}]`.
2. Флаг `enable_thinking` определяет, будет ли добавлена внутренняя цепочка рассуждений (для сложных запросов).

### Шаг 7. Генерация ответа
1. Засекается время начала.
2. Вызывается `pipe.generate()` с параметрами из `GenerationConfig`.
3. OpenVINO выполняет инференс, возвращая сгенерированный текст.
4. Вычисляется латентность.

### Шаг 8. Асинхронное обновление профиля L1
1. В фоновом потоке запускается `ActivationProfiler.update()`.
2. Вычисляется **fingerprint** — среднее эмбеддингов запроса и ответа, спроецированное в 3584.
3. В графе ищется или создаётся `activation_profile` для пары `(домен, модель)`.
4. Профиль инкрементально обновляется: пересчитывается центроид, увеличивается `sample_count`, обновляется `last_accessed`.
5. Если использовалось правило маршрутизации, его статистика (`access_count`, `success_count`) также обновляется.

### Шаг 9. Возврат результата пользователю
- Возвращается словарь с текстом ответа, определённым доменом, использованными параметрами, латентностью и числом виртуальных токенов.

## Сценарий 3. Загрузка документа для контекстной работы

### Шаг 1. Вызов `load_document`
1. Пользователь передаёт текст документа и его идентификатор (например, `"quantum_intro"`).
2. Создаётся `DocumentKnowledgeLoader`.

### Шаг 2. Создание временной сессии в графе
1. Генерируется `session_id = "doc_quantum_intro_1712000000"`.
2. Вызывается `graph.begin_temp_session(session_id)` — инициализируется список для хранения ID временных узлов.

### Шаг 3. Разбиение документа на параграфы
1. Текст разбивается по двойным переводам строк (`\n\n`).
2. Пустые параграфы отбрасываются.

### Шаг 4. Создание временных узлов для каждого параграфа
1. Для каждого параграфа:
   - Вычисляется эмбеддинг через `SentenceTransformer`.
   - Эмбеддинг проецируется в 3584.
   - Вызывается `graph.add_temp_node()` с типом `temp_paragraph`, эмбеддингом и метаданными (индекс, длина).
   - ID узла добавляется в список сессии.
2. Параграфы связываются последовательными рёбрами `next` (для сохранения порядка).

### Шаг 5. Регистрация временных узлов в гибридном токенизаторе
1. `HybridTokenizer.add_temporary_nodes()` проходит по всем узлам сессии.
2. Для каждого узла извлекается текст, создаётся виртуальный токен с новым ID (продолжая нумерацию от 151936).
3. Эмбеддинг узла сохраняется в `virtual_tokens`.
4. Текст добавляется в Aho‑Corasick автомат.
5. После добавления всех узлов автомат перестраивается.

### Шаг 6. Возврат session_id
- Идентификатор сессии возвращается пользователю для последующих запросов.

### Шаг 7. Обработка запросов по документу
- Когда пользователь задаёт вопрос, гибридный токенизатор автоматически находит совпадения с временными параграфами.
- Найденные параграфы вставляются как виртуальные токены, а их эмбеддинги могут быть использованы в префиксе.
- Модель получает прямой доступ к релевантным частям документа без необходимости держать весь текст в контекстном окне.

### Шаг 8. Завершение работы с документом
- Вызов `unload_document(session_id)` удаляет временные узлы из графа и токенизатора, освобождая память.

## Сценарий 4. Батчевая обработка огромного документа

### Шаг 1. Вызов `process_large_document`
1. Пользователь передаёт полный текст (например, книга) и желаемую задачу (`summarize`).

### Шаг 2. Чанкинг с сохранением структуры
1. `FractalChunker` разбивает текст на параграфы и группирует их в чанки, не превышающие `max_chunk_tokens`.
2. Каждому чанку присваивается порядковый номер `seq`.
3. В начало и конец чанка добавляются **структурные токены**:
   - `<|doc_start id=...|>`
   - `<|seq N|>`
   - `<|doc_end|>`
4. Чанки возвращаются списком.

### Шаг 3. Пакетная обработка
1. `BatchScheduler` сортирует чанки по длине для оптимального заполнения батчей.
2. Чанки разбиваются на батчи размером `batch_size` (например, 8).
3. Для каждого батча запускается параллельная генерация через `ThreadPoolExecutor` (каждый поток вызывает `pipe.generate()` для своего промпта).
4. Результаты собираются в список `ChunkResult` с привязкой к ID чанка.

### Шаг 4. Агрегация результатов
1. `StructuralAggregator` объединяет ответы всех чанков.
2. При стратегии `map_reduce` все промежуточные результаты конкатенируются и могут быть поданы на финальную суммаризацию.
3. При иерархической стратегии учитываются структурные токены для восстановления дерева разделов и рекурсивного слияния.

### Шаг 5. Возврат итогового результата
- Пользователь получает суммаризацию всего документа, время обработки и метаданные о структуре.

## Сценарий 5. Обучение LoRA‑адаптера для домена

### Шаг 1. Вызов `train_lora_for_domain`
1. Указывается домен (например, `"астрофизика"`).

### Шаг 2. Извлечение данных из графа
1. `FractalLoRATrainer` ищет в графе все факты, связанные с концептом домена.
2. Формируются пары «инструкция – ответ»: *«Расскажи о тёмной энергии» → «Тёмная энергия — это...»*.

### Шаг 3. Дообучение модели
1. Загружается базовая модель `RuadaptQwen3-4B` в формате HuggingFace.
2. Применяется конфигурация LoRA (`r=16`, `alpha=32`, `target_modules=["q_proj","k_proj","v_proj","o_proj"]`).
3. Запускается обучение на сформированном датасете (обычно 2–3 эпохи).

### Шаг 4. Сохранение адаптера
1. Адаптер сохраняется в safetensors в директорию `./lora_adapters/{domain}_lora`.
2. В дальнейшем он будет автоматически загружаться при запросах к этому домену.
